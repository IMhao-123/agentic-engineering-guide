## 模式 1：并行代理工作流

### Willison 的日常模式

```
同时运行 3 个 Claude Code 实例：

终端 1：探索代理
  ↓ 研究代码库、查找问题

终端 2：实现代理
  ↓ 编写代码、运行测试

终端 3：审查代理
  ↓ 代码审查、安全检查
```

### 关键点

```
1. 代理之间不能冲突
   - 不能同时编辑同一文件
   - 使用 Git 分支隔离

2. 需要共享上下文
   - 数据库 schema 定义
   - API 契约
   - 测试框架

3. 最后阶段：人工整合和审查
```

### 如何实现

```bash
# 终端 1：研究问题
git checkout -b investigate/issue-123
claude code
# 指令：研究这个 bug 的根本原因，不要修改代码，只报告发现

# 终端 2：实现修复（同时）
git checkout -b fix/issue-123
claude code
# 指令：基于规范实现修复，包含测试

# 终端 3：编写测试（同时）
git checkout -b test/issue-123
claude code
# 指令：为这个修复编写集成测试

# 完成后人工合并
```

## 模式 2：模型分层（成本优化）

```
任务               推荐模型        成本/调用
─────────────────────────────────────────
代码库探索         Claude Haiku    $0.08
常规编码           Claude Sonnet   $0.60
关键架构决策       Claude Opus     $3.00
最终安全审查       GPT-4/Opus      $3.00
```

### 实施

```bash
# 配置模型选择
export EXPLORATION_MODEL=claude-haiku
export CODING_MODEL=claude-sonnet
export REVIEW_MODEL=claude-opus

# 根据任务自动选择
if is_exploration_task; then
  use $EXPLORATION_MODEL
elif is_critical_review; then
  use $REVIEW_MODEL
fi
```

## 模式 3：CI/CD 集成

### GitHub Actions 集成

```yaml
name: Agentic Development

on: [push, pull_request]

jobs:
  agent-coding:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Run tests
        run: npm test
      
      - name: Check coverage
        run: |
          npm run test:coverage
          # 确保覆盖率 > 80%
          npx nyc check-coverage --lines 80
      
      - name: Security scan
        run: npm audit --audit-level=moderate
      
      - name: Lint
        run: npm run lint
```

## 模式 4：监控和可观测性

### 追踪 Agent 的行为

```bash
# 启用详细日志
claude code --log-level debug

# 结果输出到
~/.claude/logs/session-*.json
```

### 关键指标

```markdown
## 监控这些指标

1. Token 使用
   - 每个任务的 token 数
   - 总体趋势
   - 预算告警

2. 代码质量
   - 测试通过率
   - Bug 率（按代理生成代码）
   - 代码审查时间

3. 成本
   - 每个功能的成本
   - 成本 vs 时间节省

4. 安全
   - 权限违反次数
   - 数据泄露事件
```

## 失败恢复

### 代理出错时

```
1. 立即停止代理
   Ctrl+C

2. 回滚所有更改
   git reset --hard HEAD~1

3. 分析错误
   查看代理日志理解发生了什么

4. 修正规范
   在 SPEC.md 中添加相关约束

5. 重新启动代理
   run agent with updated spec
```

### 数据出现问题

```
如果数据被损坏：

1. 从备份恢复
   restore_from_backup(timestamp)

2. 审查所有代理操作
   agent_audit_log

3. 更新权限
   审查代理是否有不该有的权限

4. 通知团队
```

## 最后建议

> **记住最重要的：代理是工具。人类负责。**

```
好的工程师 + 好的代理 = 10x 生产力
坏的工程师 + 好的代理 = 10x 灾难
```

确保你始终：
- ✅ 理解代码（Comprehension）

- ✅ 能够教新人

不要成为"只会提示的开发者"。
