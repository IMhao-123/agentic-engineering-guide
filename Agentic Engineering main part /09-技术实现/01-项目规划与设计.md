# 01-项目规划与设计：规范是成功的 70%

> **时间：** 45-60 分钟
> **来源：** Addy Osmani《How to write a good spec for AI agents》(2026.01.13) + GitHub 2,500+ agent files 研究
> **核心观点：** 好规范决定了 70% 的成功率。而规范中**测试策略**是区分 Agentic Engineering 与 Vibe Coding 的关键。

---

## ⚠️ 最大的陷阱：直接开始写代码（Vibe Coding 模式）

**Vibe Coding 的错误流程：**
```
需求 → "写代码吧" → 代码生成 → 一团糟 → 反复修改 → 技术债务
```

**Agentic Engineering 的正确流程：**
```
需求 → 规范（包含测试策略） → 代码生成 → 测试验证 → 审查 → 可维护的代码
         ↑___________________________|
              （代理循环迭代）
```

> GitHub 分析显示：最成功的代理配置文件都遵循一个模式 — **有清晰的规范 + 明确的测试策略**。

---

## 第一步：理解 Agentic Engineering 的规划原则

### 先规划，再编码

> "Before prompting anything, you write a design doc or spec—sometimes with AI assistance. You break the work into well-defined tasks. You decide on the architecture. **This is the part vibe coders skip, and it's exactly where projects go off the rails.**"
> — Addy Osmani

**关键洞察：**
- 规格说明是你审查代理输出的**标准**——没有标准就无法审查
- 测试策略必须在编码前确定——这是与 Vibe Coding 的根本区别
- 架构决策必须由人做出——代理只负责实现

---

## 第二步：架构和技术栈决策（15 分钟）

### 代理需要知道的 6 个核心区域

这来自 GitHub Copilot 对 2,500+ agent 配置文件的分析。你的规范**必须**清楚地覆盖这 6 个：

#### **1. 命令（Commands）**：代理能运行什么

```markdown
## 构建和运行

# 本地开发
npm install              # 安装依赖
npm run dev             # 启动本地服务（http://localhost:3000）
npm test                # 运行所有测试
npm run build           # 生产构建
npm run lint --fix      # 自动修复代码风格

# 数据库
npm run migrate         # 运行 migration
npm run seed            # 填充测试数据
```

> 为什么这么重要？**代理会常常引用这些命令**。模糊的命令名字导致代理瞎试。

#### **2. 测试策略（Testing）—— 这是 Agentic Engineering 的核心**

```markdown
## 测试策略（必须在编码前确定）

### 测试框架
- 单元测试：Jest（覆盖率必须 > 80%）
- 集成测试：Supertest（API 测试）
- E2E 测试：Cypress（关键用户流程）

### 测试文件位置
- src/**/__tests__/*.test.ts  (单元测试)
- src/**/*.integration.test.ts (集成测试)
- e2e/**/*.cy.ts   (端到端测试)

### 必须测试的边界情况（代理容易遗漏的）
- 输入验证：null, undefined, empty string, 超长输入（10,000 字符）
- 安全性：SQL 注入尝试、XSS 尝试、特殊字符
- 并发：100 个同时请求
- 权限边界：用户 A 不能看到用户 B 的数据
- 错误处理：数据库连接失败、网络超时

### TDD 工作流程
1. 先写测试（定义期望行为）
2. 运行测试（应该失败）
3. 实现代码让测试通过
4. 重构并确保测试仍通过
```

> **关键：** 没有测试策略，代理会"愉快地宣称完成"但实际代码是坏的。测试是区分 Agentic Engineering 与 Vibe Coding 的核心。

#### **3. 项目结构（Project Structure）**：代码住在哪

```markdown
## 目录结构

```
src/
  ├── components/      # React 组件
  │   ├── TaskForm.tsx
  │   ├── TaskList.tsx
  │   └── UserAvatar.tsx
  ├── pages/          # 页面路由
  │   ├── Dashboard.tsx
  │   └── Settings.tsx
  ├── api/            # API 调用
  │   └── taskService.ts
  ├── types/          # TypeScript 类型
  │   └── Task.ts
  └── utils/          # 工具函数
  
__tests__/           # 测试文件
  ├── unit/          # 单元测试
  ├── integration/   # 集成测试
  └── e2e/          # 端到端测试

db/
  ├── migrations/     # SQL migrations
  └── schema.sql
```

#### **4. 代码风格（Code Style）**：你想要什么样的代码

```markdown
## 代码规范

### 命名约定
- 组件：PascalCase（TaskForm.tsx）
- 函数/变量：camelCase（getTasksForUser）
- 常量：UPPER_SNAKE_CASE（MAX_TASKS_PER_USER = 100）

### 示例代码
你想要的代码应该看起来这样：

```typescript
// ✅ 好的
async function createTask(userId: string, title: string): Promise<Task> {
  if (!title.trim()) throw new Error("Task title cannot be empty");
  
  const task = await db.task.create({ userId, title });
  return task;
}

// ❌ 不要
function ct(u, t) { return db.c({u, t}); }
```

### 其他规范
- 使用 TypeScript，不要 any 类型
- 添加 JSDoc 注释到导出函数
- 单行代码不超过 100 字符
```

#### **5. Git 工作流（Git Workflow）**：代理如何提交

```markdown
## Git 规范

### Commit 消息
格式：`<type>(<scope>): <subject>`

例子：
- ✅ `feat(task): add task priority levels`
- ✅ `fix(api): prevent duplicate task creation`
- ✅ `test(auth): add login integration tests`
- ❌ `asdf`, `fix stuff`, `update`

### 分支策略  
- 主分支：main（生产就绪）
- 开发分支：develop
- 功能分支：feature/task-prioritization

### PR 仓库
- 自动测试必须通过才能合并
- **所有代码必须经过人工审查**
```

#### **6. 边界（Boundaries）：什么绝对禁止**

```markdown
## 代理约束

### ✅ 始终做（无需确认）
- 运行测试、修复失败的测试
- 遵循代码风格规范
- 提交有意义的 commit 信息
- 更新相关文档和注释
- **每次修改后运行测试验证**

### ⚠️ 先询问，再做（需要我确认）
- 修改数据库 schema
- 添加新的 npm 依赖
- 改变 API 接口
- 修改权限/安全相关代码

### 🚫 绝对禁止（不能做）
- 提交 API 密钥、密码、敏感数据
- 修改 .env（环境配置）
- 删除现有的测试用例（除非替换更好的）
- 向 main 分支直接推送（必须 PR）
- 使用外部服务的高成本 API（如 Claude 的高级模式）
- **未经审查就合并代码**
```

## 第三步：完整规范模板（10 分钟）

把上面的都整合到一个 `SPEC.md` 文件。这个文件是**你和代理的契约**：

```markdown
# Team Task Manager - 完整规范

## 1. 项目概述
[产品简介：为谁解决什么问题，成功指标]

## 2. 技术栈
- 前端：React 18 + TypeScript + Vite
- 后端：Node.js + Express
- 数据库：PostgreSQL
- ORM：Prisma
- 测试：Jest + Cypress
- 部署：Docker + Vercel/Heroku

## 3. 核心功能（MVP）
- [ ] 用户注册/登录（JWT 认证）
- [ ] 创建任务（标题、描述、优先级、截止日期）
- [ ] 视图：我的任务、分配给我的、所有任务
- [ ] 标记完成/未完成
- [ ] 删除任务（软删除）
- [ ] 分配任务给其他用户

## 4. 数据模型
```
User {
  id: UUID
  email: string (unique)
  password: hashed
  name: string
  createdAt: timestamp
}

Task {
  id: UUID
  title: string (非空)
  description: string (可空)
  status: "todo" | "in_progress" | "done" | "archived"
  priority: "low" | "medium" | "high"
  dueDate: date (可空)
  assignedTo: User (可空)
  createdBy: User (外键)
  createdAt: timestamp
  updatedAt: timestamp
}
```

## 5. API 端点（示例）
```
POST /api/auth/register     # 注册
POST /api/auth/login        # 登录
GET  /api/tasks             # 列表（支持过滤）
POST /api/tasks             # 创建
PATCH /api/tasks/:id        # 更新
DELETE /api/tasks/:id       # 删除（软删除）
```

## 6. 测试策略（Agentic Engineering 核心）

### 测试框架
- 单元测试：Jest（覆盖率必须 > 80%）
- 集成测试：Supertest
- E2E 测试：Cypress

### 必须测试的场景
- **输入验证**：null, undefined, empty, 超长输入
- **安全性**：SQL 注入、XSS 尝试
- **权限边界**：用户 A 不能看到用户 B 的数据
- **并发**：100 个同时请求
- **错误处理**：数据库连接失败、网络超时

### TDD 工作流程
1. 先写测试定义期望行为
2. 运行测试（应该失败）
3. 实现代码让测试通过
4. 重构并保持测试通过

## 8. 项目结构
[见上面的第 3 步]

## 9. 代码风格
[见上面的第 4 步]

## 10. Git 工作流程
[见上面的第 5 步]

## 11. 边界和约束
[见上面的第 6 步]

## 12. 成功标准（如何知道完成了？）
- [ ] 所有 npm test 通过，覆盖率 > 80%
- [ ] 所有集成测试通过
- [ ] 所有 Cypress E2E 测试通过
- [ ] 没有 TypeScript 类型错误

## 第四步：从规范到代码（提示代理）

现在你有了好规范，这是最终的提示：

```
I have created a specification for a project. 

Here is the SPEC.md:
[粘贴你完整的 SPEC.md]

Now, please follow Agentic Engineering principles:

1. READ - Thoroughly read and understand the spec
2. PLAN - Create a detailed implementation plan
   - What tests will you write first?
   - What are the dependencies?
   - What might be tricky?
3. CLARIFY - Ask me any ambiguities or conflicts you find
4. TEST-FIRST - Write tests before implementation (TDD)
5. IMPLEMENT - Write code to pass tests
6. VERIFY - Run all tests and ensure they pass

Before writing any code, show me the plan and ask for confirmation.
Remember: This is Agentic Engineering, not Vibe Coding. 
Tests first, then implementation, then verification.
```

> **关键：** 不是直接让代理写代码，而是**先让代理理解规范，先写测试，再实现**。

## 📋 检查清单

完成规范后，问自己：

- [ ] 初学者读了能明白要做什么吗？
- [ ] 有没有包含 6 个核心区域？（命令、**测试**、结构、风格、Git、边界）
- [ ] **测试策略是否明确了边界情况测试？**
- [ ] 有具体例子吗？（而不是抽象的描述）
- [ ] 有清晰的成功标准吗？（包括人工审查）
- [ ] 有三层约束吗？（Always/Ask/Never）
- [ ] **是否强调了"审查每一行代码"？**

## 🎯 真实案例对比

### ❌ Vibe Coding 的坏规范导致的结果

```
客户需求："请用 React 做一个任务管理应用"

代理做的：
- 创建复杂的国际化系统（客户没要）
- 用了 5 个不同的 UI 库（没有统一风格）
- 没有测试（Vibe Coding 不关心）
- 花了 2000 token（本来需要 500）
- 最后还是需要人工重做 40%

结论：浪费时间、成本高、技术债务
```

### ✅ Agentic Engineering 的好规范导致的结果

```
规范："MVP 只需要任务 CRUD、登录、分配。技术栈 X。测试覆盖率 > 80%。"

代理做的：
- 精确符合规范要求
- 包含完整测试套件（TDD 流程）
- 代码风格一致
- 花了 600 token
- 一次成功，无需大改
- 代码可维护，新人能理解

结论：快速、准确、成本低、可维护
```
