# 00-快速启动：20 分钟体验 Agentic Engineering

> **时间：** 20 分钟
> **目标：** 理解 Agentic Engineering 与 Vibe Coding 的本质区别：测试驱动、严格审查、规划先行
> **工具：** Claude Code（推荐）或 Codex CLI
> **项目：** 一个简单的密码强度检查工具

---

## 🎯 核心原则：Agentic Engineering ≠ Vibe Coding

在开始之前，你必须理解这个根本区别：

| 维度 | Vibe Coding | Agentic Engineering |
|------|-------------|---------------------|
| **核心态度** | 盲目接受所有输出 | 严格审查每一行代码 |
| **测试** | 无或极少 | **测试是核心使能条件** |
| **规划** | 直接开始编码 | 先写规格说明 |
| **审查** | 不审查 | 以人类 PR 标准审查 |
| **产出** | 一次性原型 | 生产级软件 |

> **"测试是 Agentic Engineering 与 Vibe Coding 的最大区别"** — Addy Osmani

---

## 第一步：启动代理（2 分钟）

**打开你的编码代理：**

```bash
# 如果用 Claude Code
claude code

# 或 Codex CLI
codex

# 或 Gemini CLI  
gemini
```

进入一个新的空目录：

```bash
mkdir password-checker
cd password-checker
```

---

## 第二步：定义目标与成功标准（3 分钟）

> **关键点：** Agentic Engineering 的第一步是定义**清晰的、可验证的目标**——不是"写代码"，而是"通过测试"。

粘贴这个提示到代理中：

```
我需要创建一个命令行工具，用于检查密码强度。

目标：
- 输入一个密码
- 检查长度（至少 8 字符），大小写混合，数字，特殊字符
- 返回强度分数（0-100）和具体建议

**成功标准（这是关键）：**
- [ ] 创建 password_checker.py 文件
- [ ] 实现函数 check_password(password) → dict
- [ ] 写 5 个单元测试用例，包括边界情况（空输入、None、超长密码）
- [ ] **运行测试，全部通过**

**工作方式：**
1. 先写测试（测试驱动开发）
2. 然后实现函数让测试通过
3. 运行测试验证

立即开始：先展示你的测试计划。
```

---

## 第三步：观察代理的规划阶段（3 分钟）

代理会做什么：
1. 展示测试计划（应该包括哪些测试用例）
2. 提出实现方案
3. 要求确认才能开始

> **重要：** 如果代理有 Plan Mode，**先进入 Plan Mode**：
> ```
> Shift+Tab (Claude Code)
> # 或要求进入"只读模式"
> ```

**你要检查：**
- [ ] 测试计划是否包含边界情况（空字符串、None、超长输入）？
- [ ] 是否考虑了特殊字符和注入攻击？
- [ ] 成功标准是否清晰可验证？

---

## 第四步：测试驱动开发循环（5 分钟）

让代理按照 TDD 流程工作：

```
按照测试驱动开发流程：

1. 先写测试文件 test_password_checker.py，包含 5 个测试用例
2. 运行测试，确认它们都失败（红色）
3. 实现 password_checker.py 让测试通过（绿色）
4. 运行测试确认全部通过
5. 如果有边界情况遗漏，补充测试并修复

测试必须包括：
- 密码过短（< 8 字符）
- 没有大小写混合
- 没有数字
- 完全强的密码
- 空输入和 None 输入的处理
```

---

## 第五步：代理循环与验证（5 分钟）

代理会：
1. ✅ 生成测试
2. ✅ 运行测试（应该失败）
3. ✅ 实现代码
4. ✅ 再次运行测试（应该通过）
5. 🎉 报告完成

在这个阶段，**你注意什么**：

- [ ] 代理是否运行了测试？
- [ ] 测试是否真的通过了？
- [ ] 检查 `check_password()` 的逻辑是否合理
- [ ] 边界情况是否都处理了？

---

## 第六步：人工审查（最关键的步骤）（2 分钟）

> **不要跳过这一步。这是 Agentic Engineering 的核心。**

代理完成后，**你自己手动读一遍代码**：

```python
# 打开 password_checker.py，检查：

1. 函数签名是否清晰？
def check_password(password):
    # 它返回什么数据结构？dict? tuple?

2. 逻辑是否真的正确？
   # 密码 "AAaa11!!__" 会得到什么分数？

3. 有没有明显的 Bug?
   # 如果输入 None 会怎样？
   # 如果输入空字符串？
   # 如果输入 1000 个字符？

4. 测试够全面吗？
   # 有没有遗漏的情况？
```

**审查清单：**
- [ ] 我能解释这个函数做什么吗？
- [ ] 我能向同事解释这段代码吗？
- [ ] 如果代码出错，我能调试吗？
- [ ] 这是最好的实现方式吗？

> **如果任何一个是 NO，要求代理解释或改进。**

---

## 📊 你学到了什么？

完成这个 20 分钟的练习后，你应该理解：

✅ **1. 测试驱动是核心**
- 不是"给我写代码"，而是"先写测试，再让测试通过"
- 测试是代理知道自己是否完成目标的唯一方式

✅ **2. 人工审查不可跳过**
- 代码"通过测试"不等于"正确"
- 你必须能向他人解释代码做什么

✅ **3. 目标要具体、可验证**
- "让测试通过"比"改进代码"好一万倍
- 没有验证标准，代理会"愉快地宣称完成"但实际代码是坏的

✅ **4. Agentic Engineering 需要工程纪律**
- 规划 → 测试 → 实现 → 审查 → 验证
- 这与 Vibe Coding 的"直接生成代码"完全不同

---

## 🚨 常见问题

**Q: 代理生成的代码有 Bug，我应该怎么办？**

A: **不要 fix 后直接提交。** 要求代理修复：
```
这个函数有问题：当输入为 None 时会崩溃。
请修复并添加处理 None 的测试用例。
然后运行测试验证修复。
```

**Q: 我不懂代理生成的代码**

A: **这是最重要的警告信号。** 停下来：
1. 让代理用简单语言解释代码
2. 在代码中添加注释
3. **理解不了就不要用** — 这是 Agentic Engineering 的底线

**Q: 为什么必须先写测试？**

A: 因为：
1. 测试定义了"完成"的标准
2. 没有测试，代理不知道自己是否做对了
3. 测试让你能安全地重构和修改
4. **这是与 Vibe Coding 的根本区别**

## 🎬 测试你的理解

这个练习完成后，问自己：

1. **"为什么必须先写测试再写实现？"** 
   - 答：测试定义了完成标准，让代理能自我验证

2. **"为什么必须手工审查代码？"**
   - 答：测试不能发现所有问题，你必须理解并能为代码负责

3. **"如果代理的代码有 Bug，谁的责任？"**
   - 答：**你的**。代理只是工具。你要确保代码正确。

4. **"Agentic Engineering 和 Vibe Coding 的最大区别是什么？"**
   - 答：**测试**。Vibe Coding 没有测试，Agentic Engineering 以测试驱动。
