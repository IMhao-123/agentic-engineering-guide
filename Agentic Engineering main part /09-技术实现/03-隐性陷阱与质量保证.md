# 03-隐性陷阱与质量保证：为什么测试是核心

> **来源：** 
> - Anthropic《How AI Impacts Skill Formation》研究
> - Addy Osmani《Comprehension Debt》(2026.03.14)
> - ClawSafety 论文《"Safe" LLMs, Unsafe Agents》
> - StrongDM 真实案例
> - Simon Willison《Agentic Engineering Patterns》

> **核心警告：** Agentic Engineering 与 Vibe Coding 的最大区别是**测试**。没有测试，代理会"愉快地宣称完成"但实际代码是坏的。

---

## 陷阱 0：没有测试就启动代理循环（最严重）

### 现象

```
你：实现用户认证系统

代理：✅ 完成了！代码已生成。

你：怎么验证它工作？

代理：看起来应该能用...

结果：生产环境登录失败，用户无法访问
```

### 为什么这是最大的陷阱

> "The single biggest differentiator between agentic engineering and vibe coding is testing. With a solid test suite, an AI agent can iterate in a loop until tests pass, giving you high confidence in the result. **Without tests, it'll cheerfully declare 'done' on broken code.**"
> — Addy Osmani

**测试是核心使能条件：**
- 有测试：代理能在循环中自我修正 → 可靠系统
- 无测试：代理不知道对错 → 不可靠的代码

### 解决方案：测试驱动开发（TDD）

```markdown
## 正确流程：Red/Green/Refactor

1. RED: 先写测试（定义期望行为）
   - 运行测试 → 应该失败
   
2. GREEN: 实现代码让测试通过
   - 运行测试 → 应该通过
   
3. REFACTOR: 改进代码质量
   - 运行测试 → 仍然通过
```

**给代理的指令：**
```
我们必须遵循测试驱动开发：

1. 先写测试定义期望行为
2. 运行测试确认它们失败
3. 实现代码让测试通过
4. 重构并保持测试通过

每次修改后必须运行测试验证。
```

## 陷阱 1：Comprehension Debt（理解债务）

### 现象

```
第 1 天：代理生成代码 100 行，全部通过测试，看起来很好
第 3 天：你需要改个需求，改了代码，某个不相关的地方崩溃了
第 1 周：没人能解释为什么这样设计，为什么要这行代码
第 1 月：新人加入，无法理解现有代码，所有修改都很危险
第 3 月：业务崩溃，没人知道怎么改，只好重写
```

### 数据证实

**Anthropic 2026 年研究的真实数据：**

| 场景 | 理解分数 | 差异 |
|------|-----------|------|
| 人工编写代码 | 67% | 基准 |
| 使用 AI 生成代码并审查 | 50% | **-17 个百分点** |
| 特别是调试能力 | -25% | 最严重 |

> 换句话说：**用 AI 代理写的代码，新人的理解度低 17%**。

### 为什么发生

1. **代码生成太快** - 你没时间思考
2. **看起来"干净"** - 没有明显的坏代码味道
3. **测试通过** - 虚假的信心来源
4. **隐性假设** - 代理做了"聪明"的决策，但没记录为什么

### 解决方案

**在每个代理生成的函数后立即做这件事：**

```markdown
## 代码审查：Comprehension 检查

对于每个生成的函数，问自己：

1. "如果我需要改这个函数，我能改吗？"
   - 如果不能，要求代理：添加注释解释逻辑
   
2. "如果代码出错，我能调试吗？"
   - 如果不能，要求添加日志语句和错误处理
   
3. "新人能理解这个吗?"
   - 如果不能，要求简化或分解
   
4. "这是最好的方法吗？"
   - 如果不确定，让代理提供 3 个替代方案并解释权衡
```

## 陷阱 2：速度不对称

### 现象

```
代理：生成 1000 行代码 → 5 秒
你：审查 1000 行代码 → 2 小时

结果：看不完，就合并了（错误！）
```

### 真实数据

StrongDM 报告他们的代理循环：

```
代理 1：生成代码 - 1 秒
代理 2：运行测试 - 10 秒
代理 3：验证行为 - 30 秒
```

总时间：41 秒，全部自动化。

**但如果加入人工审查？**

```
你：审查代码并确保理解 - 30-60 分钟
```

### 后果

- 你跟不上审查
- 代码中混入未审查的逻辑
- 潜在 bug 没被发现
- 安全问题滑过去

### 解决方案

**不要审查所有代码。审查关键路径。**

```markdown
## 优先审查清单

### 🔴 必须人工审查（100% 的审查时间）
- 认证/授权逻辑
- 数据库 schema 更改
- API 响应格式（影响客户端）
- 安全相关代码
- 数据处理/隐私相关

### 🟡 采样审查（10% 的审查时间）
- 业务逻辑实现
- UI 组件
- 工具函数

### 🟢 只看测试（0% 的代码审查）
- 测试本身
- 类型定义
- 样式表
```

## 陷阱 3：测试盲点

### 现象

```typescript
// 代理写的测试
test('should create task', () => {
  const task = createTask('Get milk');
  expect(task.title).toBe('Get milk');
});

// ✅ 通过了

// 但现实中
- 字段 `title` 为空时会怎样？
- 特殊字符（"<script>alert(1)"）会怎样？
- 同时创建 1000 个任务会怎样？
- 网络超时时会怎样？

// 代理都没测试！
```

### 为什么发生

代理会**测试它想到的**，但**想不到你没想到的**。

### 解决方案

**写边界情况测试，而不是正常流程测试：**

```markdown
## 测试清单：不要只测试正常情况

### ❌ 坏的测试（太简单）
- Happy path: "输入有效数据，返回成功"
- 代理会想到的测试

### ✅ 好的测试（边界和错误）
- 空输入、null、undefined
- 太长的输入（1 MB 文本）
- 特殊字符：SQL 注入尝试
- 并发：100 个同时请求
- 网络错误、超时、部分失败
- 权限边界：用户 A 不能看到用户 B 的数据
- 数据一致性：创建后立即查询
```

**给代理的指令：**

```
写测试时，关注这些场景（不是正常流程）：

1. 输入验证
   - null, undefined, empty string
   - 超长输入（10,000 字符）
   - 特殊字符和数学符号

2. 错误处理
   - 数据库连接失败
   - 网络超时
   - 权限不足

3. 并发和性能
   - 同时 100 个请求
   - 大数据集（100 万条记录）

4. 数据完整性
   - 创建后立即读取
   - 更新不会影响其他记录
```

## 陷阱 4：权限和沙箱陷阱

### 现象

```
你启动代理，没有配置权限...
代理意外地：
- 删除了 node_modules
- 推送到 main 分支
- 修改 .env 文件
- 提交了你的 API 密钥
```

### ClawSafety 研究的发现

```
40-75% 的攻击成功率，即使在"安全"的配置下

特别危险的：
- 工作区文件注入（技能文件攻击）
- 提示符注入（通过任务描述）
- 信任滥用（代理信任"来自管理员的指令"）
```

### 解决方案

**从启动开始就配置权限（不是事后诸葛）：**

```markdown
## 权限配置检查清单

### Claude Code 权限设置

**一开始就做：**
```json
{
  "permissions": {
    "deny": [
      "Bash(git push *)",
      "Bash(git reset *)",
      "Bash(rm -rf *)",
      "WebFetch(*.env*)"
    ],
    "ask": [
      "Bash(npm install *)",
      "Bash(docker *)"
    ]
  }
}
```

### 沙箱配置

**始终使用：**
```bash
# macOS
Claude Code 默认使用 Seatbelt 沙箱 ✅

# Linux
需要安装 bubblewrap：
apt install bubblewrap socat

# 验证
claude code --verify-sandbox
```

### 网络隔离

```
只允许这些域名：
- github.com (代码仓库)
- npm.org (包管理)
- api.example.com (你的 API)

禁止：
- pastebin.com (数据泄露风险)
- telegram.org (通信工具)
- 任何外部 S3 桶
```

## 质量保证框架

### Layer 1：单元测试（基础）

```markdown
## 单元测试覆盖要求

- [ ] 所有公开函数都有测试
- [ ] 覆盖率 > 80%
- [ ] 边界情况都测试了
- [ ] 错误路径都测试了
```

### Layer 2：集成测试（关键）

```markdown
## 集成测试要求

- [ ] 前后端通信正常
- [ ] 数据库操作正确
- [ ] 权限检查有效
- [ ] API 契约一致
```

### Layer 3：Scenario Testing（端到端）

来自 StrongDM 的方法：

```yaml
# scenarios.yaml
scenarios:
  - name: "新用户注册"
    steps:
      - action: "POST /api/auth/register"
        input: {"email": "user@test.com", "password": "Pass123"}
        expect_status: 201
      
      - action: "POST /api/auth/login"
        input: {"email": "user@test.com", "password": "Pass123"}
        expect: "token must be returned"
      
      - verify: "User is in database"

  - name: "任务工作流"
    steps:
      - action: "POST /api/tasks"
        input: {"title": "Buy milk", "priority": "high"}
        expect: "task_id returned"
      
      - action: "PATCH /api/tasks/:id"
        input: {"status": "completed"}
        expect: "completed_at timestamp set"
```

## 检查清单：在合并代码前

```markdown
## 上线前必须完成

### 测试
- [ ] 所有单元测试通过
- [ ] 覆盖率 > 80%
- [ ] 集成测试通过
- [ ] 边界情况都测试了

### 代码质量
- [ ] 能手动说出这段代码做什么（Comprehension）
- [ ] 审查了所有关键路径（不是全部）
- [ ] 没有 SQL 注入、XSS、权限泄露

### 安全
- [ ] 没有 API 密钥或敏感数据在代码中
- [ ] 沙箱配置正确
- [ ] 权限审查完成

### 团队准备
- [ ] 新人能理解这段代码吗
- [ ] 有适当的文档

❌ 如果任何一个是 NO，不要合并
```

---

## 核心原则回顾

Agentic Engineering 的质量保证基于三个核心原则：

1. **测试是核心使能条件**
   - 没有测试，代理无法知道自己是否正确
   - TDD 流程：Red → Green → Refactor

2. **审查每一行代码**
   - 以人类 PR 标准审查代理输出
   - 如果无法解释，不要合入

3. **理解代码（Comprehension）**
   - 代码通过测试 ≠ 代码正确
   - 你必须能为代码负责

> **记住：** Agentic Engineering = AI 执行实现，人类掌控架构、质量和正确性。
