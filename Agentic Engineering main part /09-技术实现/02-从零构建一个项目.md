# 02-从零构建一个项目：TDD 实战演练

> **时间：** 90 分钟（包括实际操作）
> **项目：** 构建一个实际可用的待办应用（前端+后端+数据库）
> **方法：** 测试驱动开发（TDD）— Red/Green/Refactor
> **核心原则：** 先写测试 → 实现通过 → 审查 → 重构

---

## 项目架构

```
待办应用（Task Manager）
├─ 前端（React + TypeScript）→ http://localhost:3000
├─ 后端（Node.js/Express）→ http://localhost:5000
└─ 数据库（PostgreSQL）
```

---

## 阶段 1：准备规范（15 分钟）

### 创建项目目录

```bash
mkdir task-manager
cd task-manager
mkdir -p backend frontend
```

### SPEC.md：包含测试策略的完整规范

```markdown
# Task Manager - 完整规范

## 产品描述
构建一个简单的待办应用，允许用户创建和管理任务。

## 核心功能
1. 用户认证（注册/登录）
2. 创建任务（标题、描述、优先级）
3. 查看所有任务
4. 标记任务完成/未完成
5. 删除任务

## 技术栈
- 前端：React 18 + TypeScript + Axios
- 后端：Node.js + Express
- 数据库：PostgreSQL
- ORM：Prisma
- 测试：Jest（后端）+ React Testing Library（前端）

## 后端 API

```typescript
POST   /api/auth/register     // 注册
POST   /api/auth/login        // 登录
GET    /api/tasks             // 获取所有任务
POST   /api/tasks             // 创建任务
PATCH  /api/tasks/:id         // 更新任务
DELETE /api/tasks/:id         // 删除任务
```

## 数据模型

```
User {
  id: UUID
  email: string (unique)
  password: hashed
  createdAt: timestamp
}

Task {
  id: UUID
  title: string (required)
  description: string
  priority: "low" | "medium" | "high"
  completed: boolean
  userId: UUID (foreign key)
  createdAt: timestamp
}
```

## 测试策略（TDD 流程）

### 测试框架
- 单元测试：Jest
- 集成测试：Supertest
- 覆盖率目标：> 80%

### 必须测试的边界情况
- **输入验证**：空标题、超长标题（> 200 字符）、特殊字符
- **安全性**：SQL 注入尝试、XSS 尝试
- **权限**：用户 A 不能操作用户 B 的任务
- **错误处理**：数据库连接失败、无效用户 ID

### TDD 工作流程
1. 写测试定义期望行为（Red）
2. 实现代码让测试通过（Green）
3. 重构并确保测试通过（Refactor）

## 边界

### ✅ 始终做（无需确认）
- **运行测试验证每次修改**
- 遵循代码风格规范
- 添加错误处理
- 写测试覆盖边界情况

### ⚠️ 先询问（需要我确认）
- 修改数据库 schema
- 添加新依赖
- 改变 API 响应格式

### 🚫 绝对禁止（不能做）
- 提交 API 密钥或敏感数据
- 使用任何外部 SaaS（除非明确许可）
- 删除测试用例（除非替换更好的）
- **未经审查就合并代码**

## 成功标准
- [ ] 后端服务启动无错误
- [ ] 前端页面能加载
- [ ] **所有测试通过（覆盖率 > 80%）**
- [ ] **人工审查完成：我能解释每一行代码**
- [ ] 用户能注册、登录、创建、完成、删除任务

7. Run tests - they should PASS (green)

PHASE 3 - REFACTOR: Improve code quality
8. Refactor while keeping tests passing
9. Ensure code coverage > 80%

Let's start with PHASE 1. Show me your test plan before writing any code.
```

## 阶段 3：PHASE 1 - RED（写测试）（20 分钟）

### 代理应该做什么

1. **创建测试文件结构**
   ```
   backend/src/__tests__/
   ├── auth.test.ts
   ├── tasks.test.ts
   └── utils.test.ts
   ```

2. **编写测试用例**（这些测试应该失败，因为还没实现）

```typescript
// backend/src/__tests__/auth.test.ts
import request from 'supertest';
import { app } from '../server';

describe('Auth API', () => {
  describe('POST /api/auth/register', () => {
    test('should register a new user with valid data', async () => {
      const response = await request(app)
        .post('/api/auth/register')
        .send({
          email: 'test@example.com',
          password: 'Password123!',
          name: 'Test User'
        });
      
      expect(response.status).toBe(201);
      expect(response.body).toHaveProperty('token');
      expect(response.body.user.email).toBe('test@example.com');
    });

    test('should reject duplicate email', async () => {
      // 先注册一个用户
      await request(app)
        .post('/api/auth/register')
        .send({ email: 'dup@example.com', password: 'Pass123!', name: 'User' });
      
      // 再次注册相同邮箱
      const response = await request(app)
        .post('/api/auth/register')
        .send({ email: 'dup@example.com', password: 'Pass123!', name: 'User2' });
      
      expect(response.status).toBe(409);
    });

    test('should reject invalid email format', async () => {
      const response = await request(app)
        .post('/api/auth/register')
        .send({ email: 'not-an-email', password: 'Pass123!', name: 'User' });
      
      expect(response.status).toBe(400);
    });

    test('should reject weak password', async () => {
      const response = await request(app)
        .post('/api/auth/register')
        .send({ email: 'test2@example.com', password: '123', name: 'User' });
      
      expect(response.status).toBe(400);
    });

    test('should reject empty fields', async () => {
      const response = await request(app)
        .post('/api/auth/register')
        .send({ email: '', password: '', name: '' });
      
      expect(response.status).toBe(400);
    });
  });

  describe('POST /api/auth/login', () => {
    test('should login with correct credentials', async () => {
      // 先注册
      await request(app)
        .post('/api/auth/register')
        .send({ email: 'login@test.com', password: 'Pass123!', name: 'User' });
      
      // 再登录
      const response = await request(app)
        .post('/api/auth/login')
        .send({ email: 'login@test.com', password: 'Pass123!' });
      
      expect(response.status).toBe(200);
      expect(response.body).toHaveProperty('token');
    });

    test('should reject wrong password', async () => {
      await request(app)
        .post('/api/auth/register')
        .send({ email: 'login2@test.com', password: 'Pass123!', name: 'User' });
      
      const response = await request(app)
        .post('/api/auth/login')
        .send({ email: 'login2@test.com', password: 'WrongPass!' });
      
      expect(response.status).toBe(401);
    });

    test('should reject non-existent user', async () => {
      const response = await request(app)
        .post('/api/auth/login')
        .send({ email: 'nonexistent@test.com', password: 'Pass123!' });
      
      expect(response.status).toBe(401);
    });
  });
});
```

3. **运行测试，确认它们失败（RED）**
   ```bash
   npm test
   # 应该看到大量 FAIL
   ```

### 你应该检查什么

- [ ] 测试覆盖了边界情况吗？（空输入、无效输入、重复数据）
- [ ] 测试是独立的吗？（不依赖执行顺序）
- [ ] 测试断言具体吗？（不只是检查 200 状态码）

## 阶段 4：PHASE 2 - GREEN（实现代码）（30 分钟）

### 让代理实现代码以通过测试

```
现在进入 PHASE 2 - GREEN。

请实现后端代码让所有测试通过：

1. 设置 Express 服务器
2. 配置 Prisma 数据库
3. 实现认证路由（register/login）
4. 实现任务 CRUD 路由
5. 添加错误处理
6. 运行测试直到全部通过

要求：
- 每次修改后运行测试
- 如果测试失败，修复直到通过
- 保持代码简洁，只做让测试通过的最少工作
```

### 代理的工作流程

1. **设置基础架构**（package.json, tsconfig.json, Prisma schema）
2. **实现控制器**（authController.ts, taskController.ts）
3. **实现路由**（auth.routes.ts, task.routes.ts）
4. **运行测试 → 修复 → 再运行**（迭代循环）
5. **直到所有测试通过**

### 你应该监督的内容

```markdown

- [ ] 要求代理解释失败原因
- [ ] 要求代理修复并重新运行测试

### 关键检查点
- [ ] 密码是否正确哈希？
- [ ] JWT token 是否正确生成？
- [ ] 错误处理是否到位？
- [ ] 权限检查是否正确？
```

## 阶段 5：PHASE 3 - REFACTOR（重构）（10 分钟）

### 重构指令

```
所有测试通过了！现在进入 PHASE 3 - REFACTOR。

请改进代码质量，同时确保所有测试仍然通过：

1. 提取重复的代码到工具函数
2. 改进错误消息
3. 添加 JSDoc 注释
4. 检查代码覆盖率（应该 > 80%）

每次重构后运行测试验证。
```

### 重构检查清单

- [ ] 重复的代码提取了吗？
- [ ] 函数是否单一职责？
- [ ] 错误处理一致吗？
- [ ] 测试仍然全部通过吗？

---

## 阶段 6：人工审查（最关键的步骤）（10 分钟）

> **不要跳过这一步。这是 Agentic Engineering 的核心。**

**审查每一行代码：**

```markdown
## 代码审查清单

### 后端代码审查

#### authController.ts
- [ ] 理解每个函数的逻辑吗？
- [ ] 密码哈希使用的是什么算法？
- [ ] JWT 的 secret 从哪里获取？
- [ ] 错误处理覆盖了哪些情况？

#### taskController.ts
- [ ] 权限检查在哪里？
- [ ] 用户 A 能操作用户 B 的任务吗？
- [ ] 如果数据库连接失败会怎样？
- [ ] 查询是否有性能问题？（N+1 查询？）

### 审查原则

1. **"我能向同事解释这段代码吗？"**
   - 如果不能，要求代理解释或添加注释

2. **"如果代码出错，我能调试吗？"**
   - 如果不能，要求添加日志和错误处理

3. **"这是最好的实现方式吗？"**
   - 如果不确定，让代理提供 2-3 个替代方案
```

✅ **你理解了代码**

---

## 🚨 失败的迹象

❌ **跳过了测试阶段**
- 直接写实现，后补测试
- 这是 Vibe Coding，不是 Agentic Engineering

❌ **不理解代码**
- 看不懂随机生成的变量名
- 不清楚为什么这样设计
- 害怕修改任何东西

❌ **测试覆盖不足**
- 只有正常流程测试
- 没有边界情况测试
- 覆盖率 < 50%

> 如果发生 ❌ 的情况，**停下来**。重做规范，或让代理逐步解释代码。
