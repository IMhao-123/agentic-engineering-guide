# 多代理协作模式

> **核心来源：**
> - Claude Code 官方文档《Sub-agents》、《Agent Teams》、《Best Practices》并行化章节
> - Anthropic《Building Effective Agents》编排者-工作者模式
> - Simon Willison《Embracing the parallel coding agent lifestyle》（2025.10.05）
> - arXiv:2603.29632《An Empirical Study of Multi-Agent Collaboration for Automated Research》

---

## 一、多代理协作的理论基础

### 1.1 为什么需要多代理

单个代理在以下场景中会遇到根本限制：

| 限制 | 原因 | 解决方案 |
|------|------|----------|
| **上下文窗口有限** | 所有工作共享同一个上下文，复杂任务容易填满 | 为每个子任务分配独立上下文 |
| **任务类型多样** | 通用代理在不同类型任务上表现不一致 | 为每种任务类型使用专门化的代理 |
| **试错效率低** | 单一代理在错误方向上浪费上下文 | 多个代理并行尝试不同方案 |
| **审查偏差** | 写代码的代理审查自己的代码会有确认偏误 | 独立的审查代理不会偏向自己写的代码 |

### 1.2 学术实证

来源：arXiv:2603.29632《An Empirical Study of Multi-Agent Collaboration for Automated Research》

该研究对比了两种多代理架构在自动化研究任务上的表现，发现存在**操作稳定性与理论深度的根本权衡**：

| 架构 | 模式 | 优势 | 劣势 |
|------|------|------|------|
| **子代理（Subagent）** | 并行探索 + 事后整合 | 高弹性、高吞吐，适合宽泛浅层优化 | 难以实现深层理论对齐 |
| **代理团队（Agent Team）** | 专家代理 + 执行前交接 | 能实现复杂的理论对齐 | 操作脆弱性更高，需要更长计算预算 |

研究建议使用**动态路由架构**，根据实时任务复杂度自适应调整协作结构。

## 二、Claude Code 的多代理体系

Claude Code 实现了从简单到复杂的多层级多代理系统。

### 2.1 第一层：子代理（Subagents）

#### 概念

子代理是运行在**独立上下文窗口**中的专门化 AI 助手。每个子代理有自己的系统提示、工具访问权限和独立权限。当主代理遇到匹配子代理描述的任务时，它将任务委托给子代理，子代理独立工作并返回摘要。

#### 子代理的核心价值

| 价值 | 说明 |
|------|------|
| **保护主上下文** | 高输出的操作（如运行测试、搜索代码库）不会污染主对话上下文 |
| **强制约束** | 通过限制工具访问来执行特定的安全策略 |
| **成本控制** | 可以将简单任务路由到更快、更便宜的模型（如 Haiku） |
| **专门化** | 通过专注的系统提示在特定领域表现更好 |
| **可复用** | 通过配置文件在不同项目间复用 |

#### 内置子代理

Claude Code 包含以下内置子代理：

| 子代理 | 模型 | 工具 | 用途 |
|--------|------|------|------|
| **Explore** | Haiku（快速、低延迟） | 只读工具（禁用 Write/Edit） | 文件发现、代码搜索、代码库探索 |
| **Plan** | 继承主对话 | 只读工具 | Plan Mode 下的代码库研究 |
| **通用（General-purpose）** | 继承主对话 | 所有工具 | 复杂的多步骤操作 |

Claude 在委托时会指定"彻底度"级别：**quick**（快速查找）、**medium**（均衡探索）、**very thorough**（全面分析）。

#### 子代理的配置

子代理通过 Markdown 文件 + YAML frontmatter 定义。以下是一个完整的代码审查子代理示例：

```markdown
---
name: code-reviewer
description: Expert code review specialist. Proactively reviews code
  for quality, security, and maintainability.
tools: Read, Grep, Glob, Bash
model: sonnet
memory: project
---
You are a senior code reviewer ensuring high standards of code quality
and security.

When invoked:
1. Run git diff to see recent changes
2. Focus on modified files
3. Begin review immediately

Review checklist:
- Code is clear and readable
- No duplicated code
- Proper error handling
- No exposed secrets or API keys
- Input validation implemented
- Good test coverage

Provide feedback organized by priority:
- Critical issues (must fix)
- Warnings (should fix)
- Suggestions (consider improving)
```

#### 子代理的持久化记忆

子代理可以通过 `memory` 字段启用跨会话的持久化记忆：

```yaml
memory: project  # 存储在 .claude/agent-memory/ 目录
```

这使子代理能够在多次会话中积累领域知识（如代码库模式、常见问题、架构决策），随使用时间推移变得越来越有效。

#### 子代理的模型选择

可以为不同子代理选择不同的模型，实现成本和能力的平衡：

| 场景 | 推荐模型 | 理由 |
|------|----------|------|
| 代码库探索 | Haiku | 快速、低成本、只需要搜索和阅读 |
| 代码审查 | Sonnet | 需要理解代码模式和最佳实践 |
| 架构决策 | Opus | 需要强推理能力 |
| 数据分析 | Sonnet | 需要理解数据和生成洞察 |

#### 子代理的上下文管理

子代理独立于主对话运行。Claude Code 文档明确指出：

> "Subagents get their own fresh context, completely separate from your main conversation. Their work doesn't bloat your context. When done, they return a summary."

这意味着子代理可以自由地读取大量文件和运行冗长命令，而不会影响主对话的上下文质量。

### 2.2 第二层：并行会话（Parallel Sessions）

#### 概念

Claude Code 支持同时运行多个独立的会话，每个会话有自己的上下文窗口和文件访问权限。

#### 运行方式

| 方式 | 说明 | 隔离级别 |
|------|------|----------|
| **Desktop App** | 多个本地会话，各有独立的 worktree | 文件系统级 |
| **Web** | Anthropic 托管的隔离 VM | 虚拟机级 |
| **Git Worktree** | 每个 worktree 是独立的目录副本 | Git 分支级 |

#### Writer/Reviewer 模式

Claude Code 文档推荐的一个实用模式——**编写者/审查者并行**：

> "A fresh context improves code review since Claude won't be biased toward code it just wrote."

| 会话 A（编写者） | 会话 B（审查者） |
|---|---|
| `Implement a rate limiter for our API endpoints` | |
| | `Review the rate limiter implementation in @src/middleware/rateLimiter.ts. Look for edge cases, race conditions, and consistency with our existing middleware patterns.` |
| `Here's the review feedback: [Session B output]. Address these issues.` | |

这种模式的优势在于：审查者使用全新的上下文，不会被编写者已经做出的决策所影响，更容易发现编写者忽略的问题。

#### 测试的并行化

| 会话 A | 会话 B |
|---|---|
| 为模块编写测试用例 | 编写使测试通过的实现代码 |

### 2.3 第三层：Agent Teams（代理团队）

#### 概念

Agent Teams 是 Claude Code 中最高级的多代理协作形式——多个代理组成团队，由一个领导代理（Lead Agent）统一协调，共享任务、消息传递和结果合并。

#### 与子代理的区别

| 维度 | 子代理（Subagents） | Agent Teams |
|------|-------------------|-------------|
| 运行环境 | 单一会话内的独立上下文 | 多个独立的会话 |
| 通信方式 | 完成后返回摘要给主对话 | 代理间可以持续通信 |
| 生命周期 | 短暂的、任务驱动的 | 可以长时间运行 |
| 适用场景 | 隔离特定操作 | 复杂的协调任务 |

#### Fan-out 模式（批量分发）

对于大量相似任务（如迁移 2,000 个文件），可以使用批量分发模式：

```bash
# 步骤1：让 Claude 列出所有需要迁移的文件
claude -p "list all Python files that need migrating"

# 步骤2：为每个文件启动独立的 Claude 实例
for file in $(cat files.txt); do
  claude -p "Migrate $file from React to Vue. Return OK or FAIL." \
    --allowedTools "Edit,Bash(git commit *)"
done
```

`--allowedTools` 限制了代理能做什么（只能编辑和 git commit），这在无人值守运行时尤其重要。

## 三、多代理协作的关键挑战

### 3.1 代码冲突

多个代理同时修改代码库的相同文件或相关文件时，会产生 Git 合并冲突。

**解决方案：**

| 策略 | 说明 |
|------|------|
| **Git Worktree** | 每个并行会话在独立的 worktree 中运行，天然隔离 |
| **文件不重叠** | 确保各代理操作的文件集不重叠 |
| **顺序化** | 对于必须操作相同文件的任务，使用串行而非并行 |

### 3.2 上下文共享

代理之间如何共享信息和上下文仍然是一个未完全解决的问题。

**Claude Code 的方案：**

| 机制 | 说明 |
|------|------|
| **返回摘要** | 子代理完成时向主对话返回摘要，主对话决定如何传递 |
| **消息传递** | Agent Teams 支持代理间的消息传递（通过 `SendMessage` 工具） |
| **共享 CLAUDE.md** | 所有会话加载相同的项目级指令 |
| **共享 Memory** | 子代理和 Agent Teams 可以访问共享的持久化记忆 |

### 3.3 结果整合

当多个代理返回结果时，需要整合这些结果。

**Claude Code 的方案：**

主代理（Claude 本身）负责整合子代理和团队返回的结果。Claude Code 文档建议：

> "Running many subagents that each return detailed results can consume significant context."

这意味着在接收子代理结果时应注意控制返回的详细程度。

### 3.4 安全

多个代理意味着更大的攻击面。每个代理都有自己的工具访问权限和可能的 MCP 连接。

**Claude Code 的方案：**

- 子代理的工具访问权限可以独立限制
- 子代理不能生成其他子代理（防止无限嵌套）
- 每个子代理可以有自己的权限模式
- Hooks 可以在子代理级别配置安全策略

## 四、实践指南：何时使用哪种多代理模式

### 4.1 决策框架

| 需求 | 推荐模式 | 理由 |
|------|----------|------|
| 将高输出的操作隔离出主上下文 | 子代理 | 保护主上下文不被污染 |
| 为特定任务使用专门化的代理 | 子代理 | 专门的提示和工具 |
| 将简单任务路由到低成本模型 | 子代理（model: haiku） | 节省成本 |
| 需要"新鲜视角"审查代码 | 并行会话 | 避免确认偏差 |
| 同时推进多个独立任务 | 并行会话或 Git Worktree | 加速执行 |
| 大量相似的文件级任务 | Fan-out 批量分发 | 最大化并行度 |
| 复杂的协调性任务 | Agent Teams | 代理间持续通信 |
| 可复用的专门化工作流 | 子代理 + Skills | 一次定义，多处使用 |

### 4.2 模式组合

实际项目中，这些模式通常是组合使用的。例如：

**探索-实现模式：**

1. 使用 Explore 子代理在独立上下文中搜索代码库（不污染主上下文）
2. 基于子代理的发现，在主对话中实现代码
3. 使用代码审查子代理审查实现（新鲜上下文）

**测试-修复循环：**

1. 会话 A：编写测试用例
2. 会话 B：编写实现代码
3. 合并后运行测试
4. 如有失败，回到步骤 1 或 2

**大规模迁移：**

1. Claude 生成需要迁移的文件列表
2. 用 Fan-out 模式为每个文件启动独立的 Claude 实例
3. 每个实例独立完成迁移和提交
4. 合并所有变更

## 五、失败处理与恢复

### 5.1 子代理的恢复

Claude Code 支持恢复已完成的子代理：

> "Resumed subagents retain their full conversation history, including all previous tool calls, results, and reasoning. The subagent picks up exactly where it stopped rather than starting fresh."

Claude 通过 `SendMessage` 工具向已完成的子代理发送消息来恢复它。

### 5.2 子代理的失败

当子代理失败时：

- **权限不足：** 可以以前台模式启动新的前台子代理重试（此时会有交互式提示）
- **任务错误：** 检查子代理的描述和系统提示是否准确
- **上下文溢出：** 子代理的自动压缩会在约 95% 容量时触发

### 5.3 Agent Teams 的失败

Agent Teams 的失败模式更为复杂：
- 领导代理分配了不合理的任务
- 工作者代理之间的消息丢失或误解
- 多个代理对同一资源的并发修改
- 整体成本超预算

目前 Agent Teams 在 Claude Code 中仍处于实验阶段（需要 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 环境变量启用），文档建议关注其发展。
