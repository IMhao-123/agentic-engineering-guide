## 一、代理循环：所有架构的底层模型

在讨论具体的架构模式之前，必须先理解所有代理系统的底层运作模型——**代理循环（Agentic Loop）**。

### 1.1 三阶段循环

根据 Claude Code 的官方技术文档，代理在执行任务时经历三个阶段：

**阶段一：收集上下文（Gather Context）**

代理通过工具获取理解问题所需的信息——读取文件、搜索代码库、查看错误日志。Claude Code 的 Explore 子代理（使用快速的 Haiku 模型）专门负责此类只读探索，且运行在独立的上下文窗口中，避免污染主对话。

**阶段二：执行行动（Take Action）**

代理基于收集到的上下文做出决策并执行操作——编辑文件、运行命令、调用 API。每次工具调用返回的结果会反馈到循环中，为下一步决策提供新信息。

**阶段三：验证结果（Verify Results）**

代理检查自己的工作是否成功——运行测试、对比输出、检查错误。这是**整个循环中最高杠杆的环节**。

这三个阶段不是线性的，而是交织在一起的。一个 bug 修复任务可能在"执行→验证"之间反复循环多次。Claude Code 的官方文档强调：

> "The loop adapts to what you ask. A question about your codebase might only need context gathering. A bug fix cycles through all three phases repeatedly. A refactor might involve extensive verification."

### 1.2 循环的驱动力

代理循环由两个核心组件驱动：

| 组件 | 作用 | 类比 |
|------|------|------|
| **模型（Models）** | 理解代码、推理任务、分解步骤、基于反馈调整 | 大脑 |
| **工具（Tools）** | 读取文件、编辑代码、运行命令、搜索 Web、类型检查 | 手和眼睛 |

Claude Code 的官方文档将代理定义为"模型 + 工具 + 执行环境"的组合：

> "Claude Code serves as the **agentic harness** around Claude: it provides the tools, context management, and execution environment that turn a language model into a capable coding agent."

### 1.3 人类在循环中的角色

人类是代理循环的一部分。Claude Code 文档明确指出：

> "You're part of this loop too. You can interrupt at any point to steer Claude in a different direction, provide additional context, or ask it to try a different approach. Claude works autonomously but stays responsive to your input."

这种设计理念是：代理自主执行，但人类始终保持控制权。这与 Anthropic 在《Building Effective Agents》中的原则一致：代理从人类命令或交互式讨论开始工作，在遇到检查点或阻碍时暂停等待人类反馈。

## 二、基础构件：增强型 LLM

Anthropic 在《Building Effective Agents》中首先定义了代理系统的基本构件——**增强型 LLM**（Augmented LLM）。

### 2.1 三大增强能力

| 增强能力 | 说明 | 实现方式 |
|----------|------|----------|
| **检索（Retrieval）** | 从外部知识库中获取相关信息 | RAG、向量数据库、代码库搜索 |
| **工具（Tools）** | 调用外部函数和 API 执行操作 | Function Calling、MCP、Shell 命令 |
| **记忆（Memory）** | 保持跨会话的上下文信息 | 会话记忆、持久化存储、Auto Memory |

当前一代的 LLM 模型能够**主动**使用这些能力——自主生成搜索查询、选择合适的工具、决定保留哪些信息。

### 2.2 工具的分类

Claude Code 官方文档将内置工具分为五大类：

| 工具类别 | 代理能做什么 |
|----------|-----------|
| **文件操作** | 读取文件、编辑代码、创建新文件、重命名 |
| **搜索** | 按模式查找文件、用正则搜索内容、探索代码库 |
| **执行** | 运行 shell 命令、启动服务器、运行测试、使用 Git |
| **Web** | 搜索网络、获取文档、查找错误信息 |
| **代码智能** | 编辑后查看类型错误、跳转到定义、查找引用 |

### 2.3 MCP 作为工具扩展标准

Anthropic 推荐 [Model Context Protocol (MCP)](https://modelcontextprotocol.io/) 作为连接外部工具和数据源的标准化方式。Claude Code 的文档进一步明确：

> "With MCP, Claude Code can read your design docs in Google Drive, update tickets in Jira, pull data from Slack, or use your own custom tooling."

MCP 使代理的工具集可以无限扩展，而不需要修改代理核心。

## 三、两类代理系统

Anthropic 在架构层面做了关键区分：

| 类型 | 定义 | 关键特征 | 类比 |
|------|------|----------|------|
| **工作流（Workflows）** | 通过预定义的代码路径编排 LLM 和工具的系统 | 可预测、一致性强 | 自动化流水线 |
| **代理（Agents）** | LLM 动态指导自身流程和工具使用的系统 | 灵活、自主决策 | 自主工人 |

这个区分的意义在于：**大多数实际需求应该用工作流解决，而不是代理。** Anthropic 反复强调：

> "Start with simple prompts, optimize them with comprehensive evaluation, and add multi-step agentic systems only when simpler solutions fall short."

**选择依据：**

| 需求特征 | 推荐架构 |
|----------|----------|
| 任务路径可预测、步骤固定 | 工作流（提示链、路由等） |
| 任务复杂度高、步骤不可预测 | 自主代理 |
| 需要两者混合 | 工作流 + 局部代理化 |

## 四、五种核心工作流模式

### 模式一：提示链（Prompt Chaining）

**原理：** 将任务分解为一系列步骤，每个 LLM 调用处理上一步的输出。可以在中间步骤设置程序化检查点（Gate）。

**适用信号：**
- 任务可以干净地分解为固定的子任务
- 每个子任务比整体任务简单得多
- 需要在中间步骤进行质量验证

**Claude Code 中的实际对应：**

Claude Code 的官方文档推荐了一个四阶段工作流，本质上就是提示链的实践：

1. **Explore（探索）：** 使用 Plan Mode，Claude 只读文件不做修改，理解问题
2. **Plan（规划）：** 创建详细的实现计划，可导出到编辑器中手动修改
3. **Implement（实现）：** 切回 Normal Mode，让 Claude 按计划编写代码并验证
4. **Commit（提交）：** 创建描述性的提交消息并创建 PR

**实际案例：**
- 生成营销文案 → 翻译为另一种语言
- 撰写文档大纲 → 检查大纲 → 基于大纲撰写文档
- Claude Code 的 Explore → Plan → Implement → Commit 流程

### 模式二：路由（Routing）

**原理：** 对输入进行分类，并将其导向专门处理该类任务的后续流程。

**Claude Code 中的实际对应：**

Claude Code 的子代理（Subagents）系统本质上是路由模式的一种实现。Claude 根据任务描述自动将工作委托给合适的子代理：

- **Explore 子代理：** 快速的只读代理，使用 Haiku 模型，用于代码库搜索和分析
- **Plan 子代理：** 用于 Plan Mode 下的代码库研究
- **通用子代理：** 处理复杂的多步骤任务
- **自定义子代理：** 用户定义的专业代理（如安全审查、数据科学分析）

子代理的创建者通过 `description` 字段告诉 Claude 什么时候应该使用这个子代理，Claude 据此做出路由决策。

**模型级路由案例：**
- 将简单查询路由到 Haiku（低成本、低延迟）
- 将复杂架构决策路由到 Opus（强推理能力）
- Claude Code 的 `/model` 命令可以在会话中切换模型

### 模式三：并行化（Parallelization）

**原理：** 让多个 LLM 同时处理一个任务，并对输出进行程序化聚合。

**两种变体：**

| 变体 | 说明 | 案例 |
|------|------|------|
| **分片（Sectioning）** | 将任务分解为独立的子任务并行执行 | 一个代理处理前端修改，另一个处理后端 |
| **投票（Voting）** | 多次运行同一任务获得多样化输出 | 多个提示分别审查代码漏洞，任一发现即标记 |

**Claude Code 中的实际对应：**

Claude Code 的官方文档提供了多种并行化模式：

**Writer/Reviewer 模式（并行会话）：**

> "A fresh context improves code review since Claude won't be biased toward code it just wrote."

| 会话 A（编写者） | 会话 B（审查者） |
|---|---|
| 实现速率限制器 | 审查速率限制器实现中的边界情况、竞态条件和一致性 |
| 根据审查反馈修复问题 | |

**Fan-out 模式（批量分发）：**

```bash
for file in $(cat files.txt); do
  claude -p "Migrate $file from React to Vue. Return OK or FAIL." \
    --allowedTools "Edit,Bash(git commit *)"
done
```

**Agent Teams 模式（自动化协调）：**

多个代理组成团队，由一个领导代理分配任务、协调工作、合并结果。每个代理有独立的上下文窗口。

### 模式四：编排者-工作者（Orchestrator-Workers）

**原理：** 中心 LLM 动态分解任务、分派给工作者 LLM、综合其结果。

**与并行化的关键区别：** 子任务不是预定义的，而是由编排者根据具体输入动态决定的。

**Claude Code 中的实际对应：**

Claude Code 的 Agent Teams 是此模式的完整实现：
- 一个领导代理（Lead Agent）协调工作
- 工作者代理（Worker Agents）各自处理分配的子任务
- 使用消息系统在代理间通信

Claude Code 还允许用户显式使用子代理进行编排：

> "Use the code-reviewer subagent to find performance issues, then use the optimizer subagent to fix them."

### 模式五：评估者-优化者（Evaluator-Optimizer）

**原理：** 一个 LLM 生成响应，另一个提供评估和反馈，循环迭代。

**适用信号：**
- 有明确的评估标准
- 迭代优化能带来可衡量的改进价值
- LLM 的响应能在反馈下改进，且 LLM 能自行提供有效反馈

**Claude Code 中的实际对应：**

Claude Code 的整个代理循环本质上就是一个评估者-优化者循环：

1. Claude 生成代码（优化者）
2. 运行测试（评估者）
3. 测试失败 → 阅读错误信息 → 修改代码 → 重新测试
4. 循环直到测试通过

Claude Code 的官方文档将"提供验证手段"称为**"你能做的最高杠杆的事情"**：

> "Include tests, screenshots, or expected outputs so Claude can check itself. This is the single highest-leverage thing you can do."

## 五、自主代理（Autonomous Agents）

当工作流模式不足以应对任务的开放性时，就需要完整的自主代理。

### 5.1 代理循环的适应性

Claude Code 文档强调代理循环不是固定的流水线，而是根据任务动态调整的：

> "A question about your codebase might only need context gathering. A bug fix cycles through all three phases repeatedly. A refactor might involve extensive verification. Claude decides what each step requires based on what it learned from the previous step, chaining dozens of actions together and course-correcting along the way."

### 5.2 实现的简洁性

> "Agents can handle sophisticated tasks, but their implementation is often straightforward. They are typically just LLMs using tools based on environmental feedback in a loop."
>
> — Anthropic

代理的底层实现通常很简单。因此，**设计工具集和其文档**是至关重要的。Anthropic 在构建 SWE-bench 代理时，花在优化工具上的时间甚至超过了优化整体提示词的时间。

### 5.3 安全与控制

自主代理需要多层安全机制：

| 层级 | 机制 | 说明 |
|------|------|------|
| **权限系统** | Claude Code 的四种权限模式 | Default / Accept Edits / Plan / Auto |
| **检查点** | 文件变更前自动快照 | 可随时回滚到任何之前的状态 |
| **沙箱** | OS 级文件系统和网络隔离 | 限制代理的访问范围 |
| **停止条件** | 最大迭代次数、自动模式回退 | 防止代理无限循环 |
| **人类在环** | 随时中断、重定向、审查 | 人类始终是最终决策者 |

Claude Code 提供了四种权限模式：

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| **Default** | 每次文件编辑和命令执行都需要确认 | 初次使用、高敏感任务 |
| **Accept Edits** | 自动接受文件编辑，命令仍需确认 | 信任代理的方向但想审查命令 |
| **Plan Mode** | 只使用只读工具，创建计划 | 探索和规划阶段 |
| **Auto Mode** | 分类器模型在后台审查操作 | 信任代理的任务，减少中断 |

## 六、模式选择决策树

Anthropic 的核心建议是**从简单开始，只在需要时增加复杂度**。以下决策树综合了 Anthropic 文档和 Claude Code 最佳实践：

```
你的任务能否通过一次 LLM 调用解决？
├── 是 → 直接使用简单提示（Simple Prompt）
└── 否 → 任务路径是否可预测？
    ├── 是 → 使用工作流（Workflow）
    │   ├── 子任务是否独立？→ 并行化（Parallelization）
    │   ├── 子任务是否需要不同处理？→ 路由（Routing）
    │   └── 子任务是否有顺序依赖？→ 提示链（Prompt Chaining）
    └── 否 → 是否需要代理自主决策？
        ├── 是 → 使用自主代理（Agent）
        └── 否 → 评估者-优化者循环

在每一步，都应通过评估来验证是否需要更复杂的模式。
```

## 七、核心设计原则

### 7.1 Anthropic 的三条原则

1. **简洁性（Simplicity）：** 在代理设计中保持简洁。"Frameworks make it easy to get started... but they often create extra layers of abstraction that can obscure the underlying prompts."
2. **透明性（Transparency）：** 明确展示代理的规划步骤。Claude Code 的 Plan Mode 和 /context 命令是透明性的具体实现。
3. **ACI（Agent-Computer Interface）：** 投入与 HCI 同等的精力来设计代理的工具接口。

### 7.2 Claude Code 的实战原则

Claude Code 官方文档从实践中提炼了额外原则：

4. **验证优先：** "The single highest-leverage thing you can do" 是为代理提供验证手段（测试、截图、预期输出）。
5. **先探索后实现：** 分离研究和实现阶段，使用 Plan Mode 避免解决错误的问题。
6. **具体而非模糊：** "The more precise your initial prompt, the fewer corrections you'll need." 引用具体文件、提及约束、指向现有模式。
7. **上下文是最稀缺的资源：** "Most best practices are based on one constraint: Claude's context window fills up fast, and performance degrades as it fills."
