# 代理循环设计

> **核心来源：**
> - Simon Willison《Designing Agentic Loops》（2025.09.30）
> - Claude Code 官方文档《How Claude Code works》与《Best Practices》
> - Anthropic《Building Effective Agents》（2024.12）

---

## 一、代理循环的定义与本质

### 1.1 核心定义

Simon Willison 对 LLM 代理的精确定义是：

> "My preferred definition of an LLM agent is something that **runs tools in a loop to achieve a goal**."

Claude Code 的官方文档从实现角度进一步细化了这个循环，将其描述为三个交织的阶段：**收集上下文 → 执行行动 → 验证结果**。每个工具调用的返回值会反馈到循环中，为下一步决策提供新信息。Claude 根据上一步学到的东西决定下一步需要什么。

### 1.2 本质：暴力求解工具

Willison 提供了一个强有力的认知框架：

> "One way to think about coding agents is that they are **brute force tools for finding solutions to coding problems**. If you can reduce your problem to a clear goal and a set of tools that can iterate towards that goal a coding agent can often brute force its way to an effective solution."

这意味着代理循环的核心价值在于：**将人类不需要亲自执行的"乏味试错"自动化了**。人类只需要定义目标和提供工具，代理在循环中自行尝试大量方案。

### 1.3 使用代理循环的技能

> "The art of using them well is to carefully design the tools and loop for them to use."
>
> — Simon Willison

代理循环的设计不是让代理更聪明，而是让代理的工作环境更清晰、更可控、更容易验证。

## 二、代理循环的四个设计要素

### 2.1 目标（Goal）

目标是代理循环的北极星。没有清晰的目标，代理会在错误方向上消耗大量资源。

**好的目标与不好的目标：**

| 维度 | 不好的目标 | 好的目标 |
|------|-----------|---------|
| 具体性 | "改进代码质量" | "将 Docker 镜像大小减少到 200MB 以下" |
| 可验证性 | "优化性能" | "将 SQL 查询响应时间降低到 100ms 以下" |
| 范围 | "修复登录问题" | "修复 src/auth/token_refresh.py 中 token 过期后用户无法登录的问题" |

Claude Code 的最佳实践文档给出了更精确的指导——**目标应该包含验证标准**：

| 之前（不好的） | 之后（好的） |
|---|---|
| "implement a function that validates email addresses" | "write a validateEmail function. example test cases: user@example.com is true, invalid is false, user@.com is false. **run the tests after implementing**" |
| "make the dashboard look better" | "[paste screenshot] implement this design. **take a screenshot of the result and compare it to the original**. list differences and fix them" |
| "the build is failing" | "the build fails with this error: [paste error]. **fix it and verify the build succeeds**. address the root cause, don't suppress the error" |

关键词是 "run the tests"、"take a screenshot"、"verify the build succeeds"——这些是代理在循环中自我判断"我是否完成了目标"的依据。

### 2.2 工具（Tools）

工具是代理与世界交互的唯一途径。工具设计的质量直接决定代理循环的效率。

**Claude Code 官方文档中的工具分类：**

| 类别 | 工具 | 代理能做什么 |
|------|------|-----------|
| 文件操作 | Read, Write, Edit | 读取、创建、修改文件 |
| 搜索 | Glob, Grep | 查找文件、搜索内容 |
| 执行 | Bash | 运行任意命令行命令 |
| Web | Fetch | 搜索网络、获取文档 |
| 代码智能 | 诊断工具（需插件） | 编辑后查看类型错误和警告 |
| 编排 | Agent（子代理） | 将任务委托给专门的子代理 |
| 交互 | AskUserQuestion | 向人类提问 |

**工具设计的关键洞察：优先使用 Shell 命令**

Willison 的经验是：

> "Rather than leaning on MCP, I find it's usually more productive to think in terms of shell commands instead. Coding agents are really good at running shell commands!"

Claude Code 的文档更进一步——它推荐通过 CLI 工具（如 `gh`, `aws`, `gcloud`）与外部服务交互：

> "CLI tools are the most context-efficient way to interact with external services."

**在 AGENTS.md 中提供工具指引：**

Willison 推荐在项目根目录放置 AGENTS.md（或 Claude Code 的 CLAUDE.md），告诉代理可用的工具和使用方法。Claude Code 的文档对此有更详细的最佳实践——CLAUDE.md 应包含代理"猜不到"的命令和规则：

| 应该包含 | 不应该包含 |
|----------|-----------|
| 代理猜不到的 Bash 命令 | 代理通过读代码能推断的信息 |
| 与默认不同的代码风格规则 | 标准语言约定 |
| 测试指令和首选测试运行器 | 详细的 API 文档（应链接到文档） |
| 常见陷阱和非显而易见的行为 | "写干净的代码"这种自明之理 |

Claude Code 文档特别警告：**CLAUDE.md 文件过长会导致代理忽略你的重要规则**。对于每个条目，问自己："删掉它会导致代理犯错吗？"如果不会，就删掉。

### 2.3 循环条件（Loop Conditions）

代理循环需要明确的启动、继续和停止条件。

**停止条件的实现方式：**

| 停止条件 | 实现机制 | 来源 |
|----------|----------|------|
| 目标达成 | 测试通过、验证成功 | Claude Code 的测试运行 + 结果检查 |
| 最大迭代次数 | `maxTurns` 配置项 | 子代理支持此配置 |
| 人类中断 | `Esc` 键中断，`Esc + Esc` 回滚 | Claude Code 的交互设计 |
| 自动模式回退 | 分类器反复阻止操作时中止 | Claude Code 的 Auto Mode |

**Claude Code 的回滚机制：**

Claude Code 在每次文件编辑前自动创建检查点。双击 `Esc` 打开回滚菜单，可以：
- 恢复对话到之前的状态
- 恢复代码到之前的状态
- 两者同时恢复
- 从某个消息点开始总结（释放上下文）

这意味着代理可以"冒险尝试"——如果不成功，回滚到之前的状态再试不同的方法。

### 2.4 反馈机制（Feedback）

反馈是代理循环能够"自我修正"的前提。反馈的质量直接决定代理循环的收敛速度。

**Claude Code 的三种反馈来源：**

| 反馈来源 | 描述 | 可靠性 |
|----------|------|--------|
| **工具输出** | 测试结果、命令输出、文件内容 | 最可靠——"真实信息"（Ground Truth） |
| **类型检查** | 编辑后立即显示的类型错误和警告 | 高可靠性 |
| **人类输入** | 中断、重定向、审查 | 最可靠但最昂贵 |

Claude Code 文档中反复强调的最关键反馈是**自动化测试**：

> "Claude performs dramatically better when it can verify its own work, like run tests, compare screenshots, and validate outputs."
>
> "Without clear success criteria, it might produce something that looks right but actually doesn't work. You become the only feedback loop, and every mistake requires your attention."

## 三、安全运行：YOLO 模式的风险与对策

### 3.1 什么是 YOLO 模式

YOLO 模式是代理循环运行在"所有操作自动批准"的状态。不同的工具对此有不同的实现：

| 工具 | YOLO 实现方式 | 名称 |
|------|-------------|------|
| Claude Code | `--dangerously-skip-permissions` | Plan Mode 之外的完全自主模式 |
| Claude Code Auto Mode | 后台分类器审查操作 | Auto Mode |
| Claude Code Accept Edits | 自动接受文件编辑 | Accept Edits Mode |

Solomon Hykes 的名言（Willison 引用）：

> "An AI agent is an LLM wrecking its environment in a loop."

### 3.2 三大关键风险

1. **恶意命令：** 删除或篡改重要文件
2. **数据泄露：** 通过提示注入攻击窃取源代码、密钥、环境变量中的凭据
3. **代理攻击：** 利用你的机器作为代理攻击其他目标

### 3.3 Anthropic 官方的安全建议

Claude Code 的官方文档提供了分层的安全方案：

**第一层：权限模式选择**

| 模式 | 安全级别 | 适用场景 |
|------|----------|----------|
| **Default** | 最高（每步确认） | 初次使用、高敏感任务 |
| **Accept Edits** | 中高（自动接受编辑，命令仍确认） | 信任代理方向但想审查命令 |
| **Plan Mode** | 完全安全（只读） | 探索和规划阶段 |
| **Auto Mode** | 中等（分类器后台审查） | 信任任务但减少中断 |
| **Bypass Permissions** | 最低 | 仅在受控环境中使用 |

**第二层：沙箱化**

Anthropic 官方推荐的无网络容器方案：

> "Use `--dangerously-skip-permissions` in a container without internet access."

具体实现：
1. 使用 Docker Dev Containers 创建隔离环境
2. 移除网络访问
3. 将网络访问锁定到受信任的主机白名单

参考实现：[github.com/anthropics/claude-code/.devcontainer](https://github.com/anthropics/claude-code/tree/main/.devcontainer)

**第三层：凭据控制**

- 为代理创建最小权限的 API 密钥
- 使用短期有效的令牌
- 在测试/预发布环境中使用凭据
- 设置支出预算上限

### 3.4 Simon Willison 的推荐方案

Willison 个人最推荐 **GitHub Codespaces** 作为代理的运行环境：

> "I like to use GitHub Codespaces for this—it provides a full container environment on-demand that's accessible through your browser and has a generous free tier too. If anything goes wrong it's a Microsoft Azure machine somewhere that's burning CPU."

### 3.5 Claude Code 的 Auto Mode

Claude Code 提供了一种介于"完全手动"和"完全自主"之间的方案——Auto Mode。一个独立的分类器模型在后台审查每个操作，阻止：
- 权限提升（Scope Escalation）
- 未知基础设施操作
- 恶意内容驱动的操作

但允许常规操作无需确认地通过。如果分类器反复阻止操作，非交互式运行会自动中止。

## 四、上下文管理：代理循环的核心约束

### 4.1 上下文窗口是最重要的资源

Claude Code 的最佳实践文档开篇就指出：

> "Most best practices are based on one constraint: **Claude's context window fills up fast, and performance degrades as it fills.**"

代理循环的每次迭代都消耗上下文空间（读取的文件、命令输出、对话历史）。当上下文接近满时：
- LLM 可能开始"遗忘"早期指令
- 决策质量下降
- 错误率上升

### 4.2 Claude Code 的上下文管理策略

| 策略 | 说明 | 触发方式 |
|------|------|----------|
| **自动压缩（Auto Compaction）** | 当上下文接近极限时自动总结对话，保留关键代码和决策 | 自动（约 95% 时触发） |
| **手动压缩** | 用自定义指令压缩对话 | `/compact Focus on the API changes` |
| **回滚并总结** | 从某个消息点开始总结，保留之前的上下文 | `Esc + Esc` → 选择消息 → "Summarize from here" |
| **清除上下文** | 完全重置上下文 | `/clear` |
| **分离探索** | 将高输出的探索任务委托给子代理 | "Use subagents to investigate..." |

### 4.3 Claude Code 的关键经验法则

**清理上下文的时机：**

> "If you've corrected Claude more than twice on the same issue in one session, the context is cluttered with failed approaches. Run `/clear` and start fresh with a more specific prompt that incorporates what you learned."

**使用子代理保护主上下文：**

> "Since context is your fundamental constraint, subagents are one of the most powerful tools available. When Claude researches a codebase it reads lots of files, all of which consume your context. Subagents run in separate context windows and report back summaries."

## 五、会话持久化与恢复

### 5.1 会话保存

Claude Code 自动将对话保存到本地。每条消息、工具调用和结果都被存储，这使得：
- 回滚成为可能（恢复对话和/或代码到之前的状态）
- 会话可以恢复（`claude --continue` 继续最近的对话）
- 会话可以分支（`--fork-session` 从某个点创建新的分支）

### 5.2 跨会话的知识持久化

| 机制 | 说明 | 来源 |
|------|------|------|
| **CLAUDE.md** | 项目级持久化指令 | 手动编写 |
| **Auto Memory** | Claude 自动保存学习到的项目模式 | 自动生成，每次会话加载前 200 行 |
| **子代理 Memory** | 子代理的跨会话持久化记忆 | 配置 `memory: project` |
| **Git** | 代码变更的版本控制 | 手动 + 代理辅助 |

### 5.3 分支并行

Claude Code 使用 Git Worktree 支持真正的并行工作——每个并行会话在独立的目录中运行，有自己的文件副本，不会互相冲突。

## 六、何时使用代理循环

### 6.1 适合的信号

Willison 的经验法则：

> "Any time you find yourself thinking 'ugh, I'm going to have to try a lot of variations here' is a strong signal that an agentic loop might be worth trying!"

Claude Code 文档列出的典型适用场景：

| 场景 | 为什么适合代理循环 |
|------|-------------------|
| **调试** | 测试正在失败，需要调查根本原因 |
| **性能优化** | 需要尝试不同的索引策略并基准测试 |
| **依赖升级** | 需要逐个升级依赖并验证兼容性 |
| **容器优化** | 需要尝试不同的基础镜像并保持测试通过 |
| **代码审查** | 需要从多个角度审查代码质量 |
| **迁移重构** | 需要修改大量文件并验证功能正确性 |

### 6.2 不适合的场景

- 任务可以通过一次 LLM 调用解决
- 没有明确的成功标准（无法验证）
- 没有可用的测试套件
- 任务涉及高度主观的判断（如产品方向决策）

### 6.3 范围缩小的策略

当任务太大时，应缩小范围或使用子代理：

> "Scope investigations narrowly or use subagents so the exploration doesn't consume your main context."
>
> — Claude Code 文档

## 七、反模式：常见的代理循环设计错误

### 7.1 厨房水槽会话（Kitchen Sink Session）

一个会话中混杂了多个不相关的任务。上下文充满了无关信息。

**修复：** 在不相关的任务之间运行 `/clear`。

### 7.2 反复纠正循环

代理做错了，你纠正它，还是错的，你再纠正。上下文中充满了失败的尝试。

**修复：** 纠正超过两次后，`/clear` 并用更好的初始提示重新开始。Willison 的原话：

> "A clean session with a better prompt almost always outperforms a long session with accumulated corrections."

### 7.3 无限探索

让代理"调查一下"但没有限定范围。代理读取了数百个文件，上下文爆满。

**修复：** 缩小调查范围，或使用子代理将探索隔离在独立的上下文中。

### 7.4 信任-验证缺口

代理产生了"看起来能用"但实际不处理边缘情况的实现。

**修复：** 始终提供验证手段（测试、脚本、截图）。如果你无法验证它，就不要发布它。
