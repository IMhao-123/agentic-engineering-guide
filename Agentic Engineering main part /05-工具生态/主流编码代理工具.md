## 一、编码代理的三种形态

在分析具体工具之前，需要先理解编码代理的三种基本形态，它们在安全模型、使用场景和工作流上有本质区别：

| 形态 | 定义 | 代码在哪里运行 | 代表工具 |
|------|------|---------------|----------|
| **终端代理（CLI）** | 在终端中运行的 TUI/命令行工具 | 本地机器 | Claude Code、Codex CLI、Gemini CLI |
| **IDE 集成代理** | 嵌入 IDE 的 AI 代理能力 | 本地机器 | Cursor、Windsurf、Copilot in VS Code |
| **云端异步代理** | 在云端运行、异步返回结果的代理 | 云端 VM | Codex Cloud、Claude Code on the Web、Google Jules |

Willison 在 2025 年 10 月的博文中描述了自己的日常工具组合——同时使用这三种形态：

> "My daily drivers are currently Claude Code (on Sonnet 4.5), Codex CLI (on GPT-5-Codex), and Codex Cloud (for asynchronous tasks, frequently launched from my phone.)"
>
> — Simon Willison《Embracing the parallel coding agent lifestyle》

## 二、Claude Code（Anthropic）

### 2.1 概述

Claude Code 是 Anthropic 于 2025 年 2 月随 Claude 3.7 Sonnet 一起发布的终端编码代理。它是第一个进入主流的"全自主"编码代理——不仅能生成代码，还能直接执行、测试和迭代。Claude Code 目前是功能最丰富的编码代理工具，也是 Agentic Engineering 实践中最常用的工具之一。

### 2.2 多平台支持

Claude Code 的一个突出特点是跨平台覆盖：

| 平台 | 形态 | 说明 |
|------|------|------|
| **Terminal CLI** | 终端代理 | 完整功能的命令行工具，功能最全 |
| **VS Code / JetBrains** | IDE 集成 | 内联 diff、@-mentions、plan review |
| **Desktop App** | 桌面应用 | 可视化 diff review、并行会话、定时任务 |
| **Web** | 云端异步 | 浏览器中运行，无需本地安装 |
| **iOS App** | 移动端 | 通过 Claude iOS App 使用 |
| **Slack** | 团队集成 | 在 Slack 中 @Claude 触发任务 |

Claude Code 的所有表面（Terminal、VS Code、Desktop、Web）连接到相同的底层引擎，因此 CLAUDE.md 文件、设置和 MCP 服务器在所有平台上通用。

### 2.3 权限与安全模型

Claude Code 提供六种权限模式，是所有编码代理中最精细的：

| 模式 | 说明 |
|------|------|
| `plan` | 完全只读 |
| `default` | 每个工具调用需确认 |
| `acceptEdits` | 编辑自动批准，命令需确认 |
| `auto` | 后台安全分类器自动批准（研究预览） |
| `dontAsk` | 未预批准的工具自动拒绝 |
| `bypassPermissions` | 跳过权限提示——**仅在沙箱中使用** |

此外，Claude Code 是唯一提供 OS 级沙箱的终端编码代理（macOS Seatbelt / Linux bubblewrap），以及唯一提供 OpenTelemetry 集成的可观测性系统。

### 2.4 上下文管理

Claude Code 的上下文工程体系是所有工具中最完整的：

| 机制 | 说明 |
|------|------|
| **CLAUDE.md** | 项目级指令文件，支持层级继承 |
| **Auto Memory** | 自动积累的学习成果，跨会话持久化 |
| **Skills** | 按需加载的上下文包 |
| **Hooks** | 确定性执行的 Shell 命令（Pre/Post ToolUse 等） |
| **MCP** | 连接外部数据源和工具的开放协议 |
| **Auto Compaction** | 上下文窗口自动压缩 |
| **/compact、/clear** | 手动上下文管理命令 |

### 2.5 多代理支持

| 层级 | 说明 |
|------|------|
| **子代理（Subagents）** | Explore、Plan、General-purpose + 自定义（YAML frontmatter 配置） |
| **并行会话** | 多个终端窗口运行不同代理 |
| **代理团队（Agent Teams）** | 多代理协作，Leader 协调 |

### 2.6 CI/CD 集成

Claude Code 支持非交互模式（`-p` 标志），可直接嵌入 CI/CD 流水线：
- GitHub Actions
- GitLab CI/CD
- 定时任务（云端/桌面）
- 计划任务（`/schedule`）

## 三、Codex CLI（OpenAI）

### 3.1 概述

OpenAI 于 2025 年 4 月发布的命令行编码代理，运行在 GPT 系列模型上。

### 3.2 核心特点

- 命令行界面，终端运行
- 代码生成和编辑
- Shell 命令执行
- 测试运行和迭代
- 支持 YOLO 模式

### 3.3 Codex Cloud

基于云端的 Codex 运行环境，提供异步任务执行能力。Willison 将其作为"日常驱动"之一使用：

> "Codex Cloud (for asynchronous tasks, frequently launched from my phone.)"
>
> — Simon Willison

Codex Cloud 的关键优势是**异步性**——你可以在手机上发起一个任务，稍后回来检查结果。对于低风险、长时间运行的任务（依赖升级、重构），这是一个理想的选择。

---

## 四、Gemini CLI（Google）

### 4.1 概述

Google 于 2025 年 6 月发布的命令行编码代理，基于 Gemini 2.5 模型。

### 4.2 核心特点

- **超大上下文窗口：** Gemini 2.5 支持高达 1M tokens 的上下文，可以同时"看到"大量代码文件
- 基于终端运行

### 4.3 Google Jules

Google 提供的云端异步编码代理，类似于 Codex Cloud，当前免费使用。Willison 将其列为异步重构任务的可选工具。

---

## 五、Cursor

### 5.1 概述

Cursor 是目前最流行的 AI-first IDE，基于 VS Code 构建。

### 5.2 核心特点

- 深度集成 AI 能力的图形化 IDE
- 内置代码补全、聊天和代理模式
- Agent 模式可以自主编辑多个文件
- 支持多种 LLM 后端

### 5.3 与终端代理的关键区别

| 维度 | 终端代理 | IDE 集成（Cursor） |
|------|---------|-------------------|
| 工作环境 | 终端/命令行 | 图形化 IDE |
| 文件访问 | 通过工具调用 | IDE 内置 |
| 调试体验 | 文本输出 | 图形化调试器 |
| 扩展性 | MCP 服务器 | VS Code 扩展生态 |
| Diff 审查 | 终端 diff | 可视化 inline diff |
| 适合用户 | 偏好终端的开发者 | 偏好 IDE 的开发者 |

Willison 偶尔使用 GitHub Codespaces 运行 VS Code 的代理模式：

> "I occasionally use GitHub Codespaces to run VS Code's agent mode, which is surprisingly effective and runs directly in my browser. This is particularly great for workshops and demos since it works for anyone with a GitHub account, no extra API key necessary."
>
> — Simon Willison

## 六、其他工具

### 6.1 Windsurf（Codeium）

与 Cursor 类似的 AI-first IDE，提供代理模式的代码编辑能力。

### 6.2 GitHub Copilot Coding Agent

GitHub 的编码代理，直接嵌入 GitHub.com 网页界面，也作为 VS Code 扩展可用。Willison 描述其为"dabbling"级别的使用。

### 6.3 OpenCode / Crush

开源终端 AI 助手，使用 `.opencode.json` 作为配置文件。

## 七、终端代理 vs IDE 代理 vs 云端代理的选型

| 需求 | 推荐 | 原因 |
|------|------|------|
| 终端重度用户 | Claude Code 或 Codex CLI | 完整功能，精确控制 |
| IDE 重度用户 | Cursor | 图形化体验 |
| 异步/长时任务 | Codex Cloud、Claude Code on the Web、Jules | 手机发起，稍后检查 |
| 演示/教学 | GitHub Codespaces + VS Code | 任何人有 GitHub 账号即可使用 |
| 需要最大上下文窗口 | Gemini CLI（1M tokens） | Gemini 2.5 的超长上下文 |
| 需要最强安全模型 | Claude Code（OS 级沙箱 + 6 种权限模式） | 唯一提供 OS 级隔离的终端代理 |
| 需要组织级管理 | Claude Code（Managed Settings + OpenTelemetry） | 唯一提供完整组织管控的终端代理 |
| 并行任务处理 | 多个终端代理实例 | Willison 的日常模式 |
| 安全沙箱执行 | Claude Code on the Web 或 GitHub Codespaces | 代码在云端 VM 中运行 |

## 八、学术评估数据

来源：arXiv:2604.01532《PHMForge》

该研究使用 Claude Sonnet 4.0、GPT-4o、Granite-3.0-8B 配合 ReAct、Cursor Agent、Claude Code 在工业任务上进行了评估：

- 即使是最高性能的配置，任务完成率也仅为 **68%**
- 系统性的失败发生在：工具编排（23% 错误排序）、多资产推理（14.9 百分点下降）

这表明编码代理仍有显著的改进空间，也进一步强调了 Agentic Engineering 中人类监督的必要性——即使是最强的代理配置，也有近三分之一的情况无法独立完成。

## 九、Willison 的并行代理实践

Willison 总结了四种有效使用并行代理的模式：

**1. 概念验证研究（PoC Research）：**
> "Can Yjs be used to implement a simple collaborative note writing tool with a Python backend?"
>
> 用代理快速构建原型，验证技术方案的可行性。代码不需要保留。

**2. "这个怎么工作的？"（How does that work again?）：**
> "If you need a reminder about how a portion of your existing system works, modern reasoning LLMs can provide a detailed, actionable answer in just a minute or two."
>
> 让代理追踪代码路径，生成详细解释。这些解释可以保存下来作为未来 Prompt 的上下文。

**3. 小型维护任务（Small maintenance tasks）：**
> "Is your test suite spitting out a warning that something you are using is deprecated? Chuck that at a bot."
>
> 修复 deprecation warnings、修复 linting 错误等低风险小任务。

**4. 精确定义和指导的实际工作：**
> "Code that started from your own specification is a lot less effort to review."
>
> 从你的详细规格说明出发，代理执行后审查成本大幅降低。

Willison 还推荐了"Send out a scout"模式（来自 Josh Bleecher Snyder 的《The 7 Prompting Habits of Highly Effective Engineers》）：

> "Hand the AI agent a task just to find out where the sticky bits are, so you don't have to make those mistakes."
>
> 给代理一个困难任务，不打算使用其代码，只是为了了解它修改了哪些文件、如何处理问题。
