## 一、MCP（Model Context Protocol）

### 1.1 概述

MCP（Model Context Protocol）是 Anthropic 于 2024 年 11 月提出的开放标准协议，为 AI 模型提供与外部工具和数据源的标准化连接方式。

**类比：** 如果说 USB 是硬件设备的标准化接口，MCP 就是 AI 模型的标准化接口。

### 1.2 核心能力

| 能力 | 说明 |
|------|------|
| **外部工具集成** | 通过标准化协议连接外部工具和服务 |
| **工具发现** | 自动发现 MCP 服务器提供的可用工具 |
| **两种连接类型** | Stdio（标准 I/O，本地工具）和 SSE（服务器推送事件，远程服务） |
| **安全控制** | Claude Code 的权限系统控制 MCP 工具的访问 |

### 1.3 实现方式

**Stdio 类型（本地工具）：**
```json
{
  "mcpServers": {
    "example": {
      "type": "stdio",
      "command": "path/to/mcp-server",
      "args": []
    }
  }
}
```

**SSE 类型（远程服务）：**
```json
{
  "mcpServers": {
    "web-example": {
      "type": "sse",
      "url": "https://example.com/mcp"
    }
  }
}
```

### 1.4 安全考量

Claude Code 允许用户配置 MCP 服务器，但 Anthropic 不管理或审核任何 MCP 服务器。Claude Code 的安全建议：

- 仅使用你自己编写的或来自可信提供者的 MCP 服务器
- 通过权限配置控制 MCP 工具的访问
- 组织可以通过 `allowManagedMcpServersOnly` 管理设置限制仅允许受批准的 MCP 服务器

### 1.5 生态系统

- Claude Code、Cursor 等主要编码代理工具均支持 MCP
- GitHub 有 MCP Registry（注册表）
- 已有大量开源 MCP 服务器实现

## 二、AGENTS.md / CLAUDE.md：项目上下文文件

### 2.1 什么是项目上下文文件

项目上下文文件是一种约定俗成的配置文件，放置在项目根目录，为编码代理提供项目特定的指令和上下文。不同工具使用不同的文件名：

| 文件名 | 工具 | 说明 |
|--------|------|------|
| `CLAUDE.md` | Claude Code | Claude Code 自动发现和加载 |
| `AGENTS.md` | 通用约定 | OpenAI Agents SDK 等多种工具支持 |
| `.cursorrules` | Cursor | Cursor 的配置文件 |
| `.opencode.json` | OpenCode / Crush | OpenCode 的配置文件 |

### 2.2 关键研究发现：自动生成 vs 手动编写

这是 2026 年初最重要的代理工程实践发现之一，来自两篇学术论文的直接碰撞。

**研究一：Lulla et al.（ICSE JAWs 2026, arXiv:2601.20404）**
- 124 个真实 GitHub PR，有/无 AGENTS.md 的配对实验
- 结果：**人工编写的 AGENTS.md 使中位运行时间减少 28.64%，输出 Token 消耗减少 16.58%**
- 注意：该研究仅测量效率，未测量正确性

**研究二：ETH Zurich（arXiv:2602.11988）**
- 四个代理，SWE-bench + 自定义基准
- 结果：**LLM 自动生成的上下文文件使任务成功率降低 2-3%，成本增加 20%+**
- 开发者编写的文件使成功率提高约 4%，但成本也增加最多 19%

**关键洞察：自动生成的内容不是没有用，而是**冗余的**。代理本来就能自己发现这些信息（通过读取代码库），给两份相同的信息只会增加噪音。**

**ETH Zurich 的进一步实验：** 当他们从代码库中删除所有文档（README、docs 文件夹等），然后测试 LLM 生成的上下文文件时，这些文件反而使性能提高了 2.7%。这证实了冗余假说——问题不在于生成的内容本身，而在于它与已有文档重复。

### 2.3 什么值得写进 AGENTS.md

ETH Zurich 的研究发现了具体的有效模式：

| 工具 | 提及时的使用频率 | 未提及时的使用频率 |
|------|-----------------|-------------------|
| `uv`（Python 包管理器） | 1.6 次/任务 | 不到 0.01 次/任务 |
| 其他仓库特定工具 | 2.5 次/任务 | 不到 0.05 次/任务 |

`uv` vs `pip` 是完美的例子——它是：
- **不可发现的：** 代理无法仅通过阅读代码推断出来
- **操作上重要的：** 它改变了代理运行的命令
- **无法猜测的：** 按惯例无法确定应该用哪个

相比之下，"这个项目使用 monorepo 结构"——代理在第一次 `ls` 就能发现。一个值得写，一个是噪音。

**实用过滤器：** 代理能自己通过阅读代码发现这个信息吗？如果能，不写。

### 2.4 锚定效应（Anchoring Effect）

Osmani 指出了一个微妙但重要的成本：

> "If your AGENTS.md mentions that you use tRPC on the backend - even just as a passing note - the model now has tRPC in context for every single prompt. If tRPC is only used in a handful of legacy endpoints and everything new runs on something else, you've just biased your agent toward the wrong pattern for the current codebase."

你在 AGENTS.md 中写的每一行，都在与代理当前任务竞争注意力。Liu et al. 的"Lost in the Middle"（2024）表明 LLM 在处理长上下文中间的信息时表现不佳。Levy et al. 显示即使内容完全相关，更长的上下文也会降低任务表现。

### 2.5 Osmani 提出的三层架构

Osmani 认为单一扁平的 AGENTS.md 文件对任何有真实复杂度的代码库都不够。他提出了三层架构：

**Layer 1：协议文件（Protocol File）**
不是一个代码库概览或风格指南。一个**路由文档**——列出可用的 persona 和何时调用、可用的 skill、可用的 MCP 连接、代理无法发现的少量核心事实。

**Layer 2：聚焦的 persona/skill 文件**
根据任务类型按需加载。一个做 UX 组件架构的代理和一个做数据管道调试的代理加载完全不同的上下文。

**Layer 3：维护子代理**
唯一的职责是保持协议文件随代码库演化而准确。

**静态文件的根本问题：** 即使写得很紧凑，静态指令也无法根据任务类型条件化。"运行测试套件"对文档变更来说是浪费 Token，对安全修复来说可能应该跳过性能优化提示。

### 2.6 ACE 框架与自动优化

**ACE 框架（arXiv:2510.04618, ICLR 2026）** 将上下文视为演化的 playbook 而非静态文件，通过 generator/reflector/curator 管道动态适应。在代理基准测试中，动态方法比静态方法高出 12.3%。

**Arize AI 的 Prompt Learning：** 不手动编写 CLAUDE.md 指令，而是使用自动化优化循环——在训练任务上运行代理，评估输出，用 LLM 反馈生成改进建议，用元提示（meta-prompting）精炼指令。结果：跨仓库测试集准确率 +5.19%，仓库内测试集 +10.87%。

**关键发现：** 帮助人类理解代码库的信息和帮助 LLM 导航代码库的信息往往是不同的。你以为"显然有用"的指令对模型可能是噪音，模型实际需要的信息你可能从未想过要写下来。

### 2.7 正确的心态

Osmani 的核心建议：

> "Think of AGENTS.md as a living document of friction you haven't fixed yet."

每一行都代表代码库中某些足以绊倒 AI 代理的东西——而这些东西可能也足以绊倒新的人类贡献者。正确的回应不是增长上下文文件，而是修复底层问题。

**实践技巧：** 从几乎空的 AGENTS.md 开始，只加一条指令："If you encounter something surprising or confusing in this project, flag it as a comment." 然后根据代理的反馈修复代码库问题，保持文件最小化。

## 四、Git 与编码代理

### 4.1 LLM 的 Git 能力

Simon Willison 指出了 LLM 在 Git 操作方面的出色能力：

> "LLMs are also fiercely competent at Git—they can navigate the history themselves to track down the origin of bugs, and they're better than most developers at using git bisect."

编码代理可以执行的高级 Git 操作：
- `git bisect` 二分查找引入 Bug 的提交
- `git log` / `git blame` 追溯变更历史和责任人
- `git diff` 分析代码变更
- 自动提交、分支管理、冲突解决
- 自动生成 commit message 和 PR description

### 4.2 推荐实践

- 代理的每次重要变更都应提交——方便回滚
- 使用分支策略隔离代理的工作——避免并行代理的代码冲突
- 利用 `git bisect` 让代理定位问题根源
- **审查代理生成的 commit message 和 PR description**——Willison 明确指出代理会生成"令人信服但可能不准确"的描述

## 五、CI/CD 集成

### 5.1 代理生成的代码纳入 CI 流水线

代理生成的代码应与传统代码通过完全相同的 CI/CD 检查：

```
代码提交 → 单元测试 → 类型检查 → Linting → 构建 → 安全扫描 → 人工审查
```

### 5.2 代理作为 CI 工具

Claude Code 支持非交互模式（`-p` 标志），可以直接嵌入 CI 流水线：
- GitHub Actions
- GitLab CI/CD
- 自动修复 CI 中发现的 linting 错误
- 自动更新依赖并验证兼容性
- 自动生成变更日志

### 5.3 定时任务

Claude Code 支持定时任务执行：
- 云端定时任务（Claude Code on the Web）
- 桌面定时任务（Desktop App）
- `/loop` 在 CLI 会话中重复执行 prompt

## 六、LSP 与代码智能

语言服务器协议（LSP）为传统 IDE 提供代码智能能力：诊断（Diagnostics）、补全（Completion）、定义跳转（Go to Definition）、引用查找（Find References）。

对于编码代理，这些能力通过不同的方式实现：
- Claude Code 内置代码理解能力（通过 Read、Grep、Glob 工具直接搜索代码库）
- Cursor 等 IDE 集成代理可以利用 VS Code 的 LSP 生态
- 终端代理通过 Shell 命令（`grep`、`rg`、`ast-grep` 等）实现类似的代码导航能力

Willison 的经验表明，编码代理在代码搜索和路径追踪方面"extremely effective"，在大型代码库中能跨数十个文件追踪代码路径。

## 七、并行代理基础设施

### 7.1 Git Worktree 隔离

Claude Code 官方文档推荐使用 Git Worktree 运行并行会话：

> "Run parallel Claude Code sessions with git worktrees"
> — Claude Code Common Workflows

每个代理在一个独立的 worktree 中工作，避免代码冲突。

### 7.2 云端隔离

对于风险较高的任务，Willison 使用异步云端代理（Codex Cloud、Claude Code on the Web），即使出问题，最坏的情况也只是源代码泄露（因为允许网络访问），不会影响本地机器。

### 7.3 Willison 的并行代理模式

| 模式 | 说明 | 风险级别 |
|------|------|----------|
| 概念验证研究 | 快速构建原型，代码不保留 | 低 |
| "这个怎么工作的？" | 让代理追踪代码路径并生成解释 | 低 |
| 小型维护任务 | 修复 warnings、linting 错误等 | 低 |
| 精确定义的工作 | 从你的详细规格出发，审查成本低 | 中 |
| "派出侦察兵" | 给代理困难任务只为了解它如何处理，不保留代码 | 低 |

Willison 目前通常有多个终端窗口同时运行不同的编码代理，使用 YOLO 模式处理他确信不会有恶意指令混入的任务。但他承认：

> "I need to start habitually running my local agents in Docker containers to further limit the blast radius if something goes wrong."
>
> — Simon Willison
