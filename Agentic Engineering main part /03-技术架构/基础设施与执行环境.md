## 一、代理执行环境的三个层次

Claude Code 的官方文档定义了三种执行环境，各有不同的安全性和灵活性权衡：

| 环境 | 代码在哪里运行 | 适用场景 |
|------|---------------|----------|
| **Local（本地）** | 你的机器 | 默认。完整的文件、工具和环境访问 |
| **Cloud（云端）** | Anthropic 托管的 VM | 无需本地安装、处理没有本地的仓库 |
| **Remote Control（远程控制）** | 你的机器，通过浏览器操控 | 在手机/平板上远程使用 |

对于 Agentic Engineering 的日常实践，**本地环境**是最常用的。但无论是哪种环境，代理的执行基础设施需求是相同的。

## 二、沙箱化：代理安全运行的基石

### 2.1 为什么沙箱是必须的

Claude Code 的安全模型基于一个核心事实：**代理有权限执行你能执行的所有操作。** 这意味着一个失控的代理可以删除文件、窃取凭据、部署恶意代码。沙箱将代理的行为限制在可控范围内。

### 2.2 Claude Code 的三种沙箱方案

#### 方案一：权限模式（Permission Modes）

Claude Code 提供四种权限模式，从最安全到最自主：

**Default 模式：**
- 文件编辑需要确认
- Shell 命令需要确认
- MCP 工具调用需要确认

**Accept Edits 模式：**
- 文件编辑自动批准
- Shell 命令仍需确认
- 适用于：信任代理的编辑方向但想审查命令执行的场景

**Plan 模式：**
- 完全只读，只能使用 Read、Grep、Glob 等工具
- 不会对系统做任何修改
- 适用于：探索和规划阶段

**Auto 模式：**
- 后台分类器模型审查所有操作
- 自动阻止：权限提升、未知基础设施操作、恶意内容驱动的操作
- 自动放行：常规的文件编辑和命令
- 如果分类器反复阻止操作，非交互运行会自动中止

**Bypass Permissions 模式：**
- 跳过所有权限检查
- 仅应在完全受控的环境中使用

**实际选择建议：**

| 工作阶段 | 推荐模式 | 理由 |
|----------|----------|------|
| 探索和规划 | Plan | 零风险了解代码库 |
| 日常开发 | Default | 安全性和效率的平衡 |
| 大量重复操作 | Accept Edits | 减少确认疲劳（确认超过 10 次后你其实不在审查了） |
| 无人值守任务 | Auto | 自动化安全审查 |

#### 方案二：OS 级沙箱（Sandboxing）

Claude Code 提供了操作系统级别的隔离：

- **文件系统隔离：** 限制代理可以访问的文件和目录
- **网络隔离：** 限制代理可以连接的网络地址

Claude Code 的文档推荐使用 Apple Container 或 Docker 容器来实现这种隔离。

#### 方案三：Git Worktree 隔离

对于并行运行多个代理的场景，Git Worktree 提供了代码仓库级别的隔离：

- 每个 worktree 是仓库的一个独立副本
- 多个代理可以同时操作不同的 worktree 而不产生文件冲突
- Worktree 使用完毕后可以自动清理（如果子代理没有做任何修改）

### 2.3 Anthropic 官方的安全推荐

Anthropic 官方文档给出的"安全 YOLO 模式"方案：

> "Use `--dangerously-skip-permissions` in a container without internet access."

具体步骤：
1. 使用 Docker Dev Containers 创建隔离容器
2. 在容器中移除网络访问
3. 将网络访问锁定到受信任的主机白名单
4. 在此容器中运行 `--dangerously-skip-permissions`

参考实现：[github.com/anthropics/claude-code/.devcontainer](https://github.com/anthropics/claude-code/tree/main/.devcontainer)

## 三、凭据管理

### 3.1 核心原则

> "Ideally we wouldn't need any credentials at all."
>
> — Simon Willison

凭据管理在代理环境中比传统开发更关键，因为代理可以自主执行命令，一个不受限制的 API 密钥可能导致意外的大量支出。

### 3.2 凭据策略

| 策略 | 说明 | 实现方式 |
|------|------|----------|
| **最小权限** | 只授予完成任务所需的最小权限 | 使用作用域受限的 API 密钥 |
| **短期有效** | 凭据有短暂的过期时间 | 使用 OIDC Token |
| **测试/预发布** | 使用非生产环境的凭据 | 为每个代理任务创建独立的测试环境 |
| **预算上限** | 设置支出限制 | 云服务的预算告警 |
| **环境变量** | 通过环境变量注入，不硬编码 | 容器环境变量或 .env 文件 |

### 3.3 Willison 的实战案例

Willison 在调试一个性能问题时创建了一个专门的隔离环境：

1. 在 Fly.io 上创建了一个独立组织
2. 设置了 $5 预算上限
3. 为该组织签发 API 密钥
4. 将密钥提供给 Claude Code

这样即使代理行为异常，损失也被限制在 $5 以内。

### 3.4 权限白名单

Claude Code 支持在 `.claude/settings.json` 中配置权限白名单：

```json
{
  "permissions": {
    "allow": [
      "Bash(npm test *)",
      "Bash(git status)",
      "Bash(git diff *)"
    ]
  }
}
```

这样代理在执行这些特定的命令时不需要人工确认，而对于其他命令仍需确认。

## 四、环境配置

### 4.1 Claude Code 的配置层级

Claude Code 的配置按以下优先级从高到低排列：

| 层级 | 位置 | 作用域 |
|------|------|--------|
| 托管设置 | Anthropic 部署 | 组织级策略，最高优先级 |
| `--agents` CLI 参数 | 当前会话 | 仅当前会话有效 |
| `.claude/settings.json` | 当前项目 | 项目级配置 |
| `.claude/settings.local.json` | 当前项目 | 个人项目配置（不提交到 Git） |
| `~/.claude/settings.json` | 用户主目录 | 所有项目的个人偏好 |
| 插件 | `.claude/plugins/` | 插件提供的配置 |

### 4.2 开发环境配置

**安装 Claude Code：**

Claude Code 支持多种安装方式：

```bash
# macOS / Linux / WSL（推荐）
curl -fsSL https://claude.ai/install.sh | bash

# Homebrew
brew install --cask claude-code

# WinGet
winget install Anthropic.ClaudeCode
```

**配置第三方模型提供商：**

Claude Code 支持通过 Anthropic Console 或第三方提供商使用不同的模型。

**配置 MCP 服务器：**

```bash
claude mcp add  # 交互式添加
```

**安装插件：**

```bash
/plugin  # 浏览和安装社区插件
```

## 五、环境变量与模型配置

### 5.1 关键环境变量

| 环境变量 | 说明 |
|----------|------|
| `ANTHROPIC_API_KEY` | Anthropic API 密钥 |
| `OPENAI_API_KEY` | OpenAI API 密钥 |
| `GEMINI_API_KEY` | Google Gemini API 密钥 |
| `CLAUDE_CODE_SUBAGENT_MODEL` | 覆盖所有子代理的模型选择 |
| `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS` | 禁用后台任务功能 |

### 5.2 模型选择策略

Claude Code 支持在不同场景使用不同的模型：

| 模型 | 特点 | 适用场景 |
|------|------|----------|
| **Haiku** | 快速、低成本、低延迟 | 代码库探索、简单任务 |
| **Sonnet** | 能力和速度的平衡 | 大多数编码任务 |
| **Opus** | 最强推理能力 | 复杂架构决策、困难调试 |

可以在会话中用 `/model` 切换，或用 `claude --model opus` 启动会话。子代理可以通过 `model` 字段使用不同的模型。
