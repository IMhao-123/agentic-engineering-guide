## 一、威胁模型：为什么编码代理的安全问题比传统工具严重得多

传统开发工具——IDE、构建系统、包管理器——都是**确定性系统**：你给它们什么输入，它们就做什么输出，行为可预测。编码代理则完全不同：它是一个**自主决策系统**，可以在你不知情的情况下执行任意命令、修改任意文件、发起任意网络请求。正如 Solomon Hykes 所说，"An AI agent is an LLM wrecking its environment in a loop"（AI 代理就是一个在循环中破坏环境的 LLM）。这个表述虽然戏谑，但精准地指出了代理安全问题的本质。

### 1.1 核心威胁来源

来源：Simon Willison《Designing Agentic Loops》及 ClawSafety 论文（arXiv:2604.01438）

Willison 总结了 YOLO 模式（即代理可以自主执行命令而无需人类确认的模式）的三大风险：

| 风险 | 说明 | 后果 |
|------|------|------|
| **恶意 Shell 命令** | 代理执行了删除、篡改文件的命令 | 代码丢失、系统损坏 |
| **数据泄露** | 通过提示注入攻击窃取源代码、密钥、环境变量中的凭据 | 知识产权泄露、账号被盗 |
| **代理攻击** | 利用你的机器作为代理攻击其他目标（DDoS、掩饰攻击来源） | 法律责任、资源滥用 |

### 1.2 ClawSafety：首个系统性编码代理安全评估

来源：arXiv:2604.01438《ClawSafety: "Safe" LLMs, Unsafe Agents》（2026 年 4 月，Bowen Wei 等）

ClawSafety 是目前最系统的编码代理安全实证研究，测试了五个前沿 LLM 作为代理骨干在 2,520 个沙箱试验中的表现。研究设计了 120 个对抗性测试场景，覆盖三个攻击维度：

- **危害领域：** 软件工程、金融、医疗、法律、DevOps
- **攻击向量：** 工作区技能文件（workspace skill files）、来自信任发送者的邮件、网页内容
- **有害行为类型：** 凭据转发、破坏性操作、数据泄露

**关键发现：**

1. **攻击成功率（ASR）在 40% 到 75% 之间**——即使是最强的模型也无法完全抵御所有攻击
2. **技能文件注入（最高信任级别）始终比邮件或网页内容更危险**——代理最信任工作区中的配置文件，因此这种注入途径的 ASR 最高
3. **安全不仅仅取决于骨干模型，还取决于完整的部署栈**——跨三个代理框架的交叉实验表明，同一个模型在不同框架下的安全表现差异显著
4. **最强模型在凭据转发和破坏性操作上保持了硬边界**，但较弱模型同时允许了这两种行为

### 1.3 提示注入（Prompt Injection）：代理面临的最独特威胁

提示注入是编码代理面临的最独特安全威胁，因为它利用了代理的核心机制——**用自然语言指令控制行为**——来对抗代理本身。

**攻击原理：** 攻击者通过代理处理的任何数据（网页内容、邮件、项目文件）嵌入恶意指令，试图覆盖代理的系统提示词和安全约束。

**攻击路径（按危险程度排列）：**

| 攻击路径 | 危险程度 | 原因 |
|----------|----------|------|
| **技能文件注入** | 最高 | 技能文件是代理最信任的数据来源（如 AGENTS.md、CLAUDE.md），代理会将其中的内容视为指令 |
| **邮件注入** | 中高 | 来自"信任发送者"的邮件，攻击者可以伪造发件人 |
| **网页注入** | 中 | 代理访问的网页可以包含隐藏的恶意指令 |

ClawSafety 的数据证实了这一排序：技能文件的攻击成功率一致高于其他两个向量。

## 二、Claude Code 的安全架构：逐层防御

Claude Code 的安全设计遵循**纵深防御（Defense-in-Depth）**原则，不是依赖单一防护层，而是在多个层面设置独立的安全机制。

### 2.1 第一层：基于权限的架构（Permission-Based Architecture）

Claude Code 默认采用**严格的只读权限**。当需要执行额外操作（编辑文件、运行测试、执行命令）时，必须请求明确的许可。

**权限分级：**

| 工具类型 | 示例 | 是否需要确认 | "不再询问"的行为 |
|----------|------|-------------|-----------------|
| 只读工具 | 文件读取、Grep、Glob | 否 | 不适用 |
| Bash 命令 | Shell 执行 | 是 | 永久性，按项目目录和命令 |
| 文件修改 | 编辑/写入文件 | 是 | 仅限当前会话 |

**六种权限模式：**

Claude Code 提供六种权限模式，从最安全到最自主：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| `plan` | 完全只读，仅分析与规划 | 探索代码库、制定方案 |
| `default` | 标准模式，首次使用每个工具需确认 | 日常开发 |
| `acceptEdits` | 自动批准文件编辑，但命令仍需确认 | 信任代理编辑方向但想审查命令 |
| `auto` | 使用后台安全分类器自动批准，研究预览版 | 需要平衡自主性和安全性 |
| `dontAsk` | 自动拒绝未预批准的工具 | 高度管控环境 |
| `bypassPermissions` | 跳过权限提示 | **仅在沙箱或容器中使用** |

**规则评估顺序：** deny → ask → allow。deny 规则始终优先。

**权限规则的精细控制：**

权限规则支持通配符匹配，可以按工具、按命令、按文件路径精确控制。例如：

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git commit *)"
    ],
    "deny": [
      "Bash(git push *)"
    ]
  }
}
```

Claude Code 甚至能感知 Shell 操作符（`&&`、`|`），因此一个 `Bash(safe-cmd *)` 规则不会意外批准 `safe-cmd && malicious-cmd`。

**一个重要的安全警告：** 试图用 Bash 权限模式约束命令参数是脆弱的。例如 `Bash(curl http://github.com/*)` 无法匹配 `curl -X GET https://github.com/...` 或 `curl -L http://bit.ly/xyz`（重定向到 GitHub）。对于可靠的 URL 过滤，Claude Code 推荐使用 WebFetch 工具的域名权限规则，或通过 PreToolUse Hook 实现自定义验证。

### 2.2 第二层：OS 级沙箱（OS-Level Sandboxing）

权限系统控制的是"Claude Code 可以使用哪些工具"。沙箱控制的是"Shell 命令可以访问哪些系统资源"。两者是**互补的安全层**。

**沙箱的技术实现：**

| 操作系统 | 沙箱技术 | 说明 |
|----------|---------|------|
| macOS | Seatbelt | 苹果内置的沙箱框架，开箱即用 |
| Linux | bubblewrap | 需要安装 `bubblewrap` 和 `socat` |
| WSL2 | bubblewrap | 与 Linux 相同 |
| WSL1 | 不支持 | bubblewrap 需要的内核特性仅在 WSL2 中可用 |

**沙箱提供的两种隔离：**

**文件系统隔离：**
- 默认可读写当前工作目录及其子目录
- 默认可读取整个计算机（除被拒绝的目录）
- 无法修改工作目录之外的文件（除非明确配置）
- 可通过 `sandbox.filesystem.allowWrite`、`denyWrite`、`denyRead`、`allowRead` 精确控制

限制在 OS 级别执行，因此适用于所有子进程命令——包括 `kubectl`、`terraform`、`npm` 等。

**网络隔离：**
- 通过沙箱外部的代理服务器控制
- 仅允许白名单中的域名
- 新域名请求触发权限提示
- 支持自定义代理（可解密、检查 HTTPS 流量）
- 限制同样适用于所有子进程脚本和程序

**两种沙箱模式：**

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| Auto-allow | 沙箱内的命令自动批准，沙箱外的命令走正常权限流程 | 减少权限疲劳 |
| Regular permissions | 所有命令都走正常权限流程，即使是沙箱内的 | 需要完全控制的场景 |

**Anthropic 官方的安全 YOLO 模式指南：**

> "Letting Claude run arbitrary commands is risky and can result in data loss, system corruption, or even data exfiltration (e.g., via prompt injection attacks). To minimize these risks, use `--dangerously-skip-permissions` in a container without internet access."
>
> — Anthropic《Claude Code Best Practices》

具体实施方法：在 Docker 容器中运行 Claude Code，将互联网访问限制为**可信域名白名单**。Anthropic 提供了一个参考实现（`github.com/anthropics/claude-code/tree/main/.devcontainer`），其中包含一个防火墙初始化脚本，将网络访问限制到指定的域名列表。

### 2.3 第三层：提示注入防护

Claude Code 包含多层提示注入防护机制：

**核心防护：**
- **权限系统：** 敏感操作需要明确的人类批准
- **上下文感知分析：** 分析完整请求以检测潜在有害指令
- **输入清洗：** 处理用户输入以防止命令注入
- **命令黑名单：** 默认阻止 `curl`、`wget` 等获取任意网络内容的命令
- **失败关闭匹配：** 未匹配的命令默认需要手动批准

**隐私保护：**
- 敏感信息的保留期限有限
- 对用户会话数据的访问受限
- 用户可控制数据训练偏好

**WebFetch 的隔离上下文窗口：** Web 获取工具使用**独立的上下文窗口**来处理网页内容，防止网页中的恶意指令注入到主对话上下文中。

**信任验证：** 首次在代码库中运行和新的 MCP 服务器需要信任验证。

**命令注入检测：** 可疑的 Bash 命令即使之前已被白名单批准，也会要求手动确认。

**网络请求批准：** 发起网络请求的工具默认需要用户批准。

### 2.4 第四层：凭据管理

> "Issue tightly scoped credentials."
>
> — Simon Willison《Designing Agentic Loops》

凭据管理是代理安全的另一个关键维度。代理可以执行你能执行的所有操作，这意味着它能访问你能访问的所有凭据。

**Willison 的凭据管理建议：**

1. **为代理创建最小权限的 API 密钥**——密钥的权限范围应恰好覆盖代理需要执行的操作
2. **使用短期有效的令牌**——限制凭据的有效期，即使泄露也能将损失控制在时间窗口内
3. **在测试/预发布环境中使用凭据**——避免暴露生产环境的密钥

**Willison 的实践案例：** 在调查 Fly.io 冷启动性能问题时，Willison 创建了一个专用的 Fly 组织，设置了 $5 的预算上限，然后生成了仅限于该组织的 API 密钥交给 Claude Code。这样即使代理失控，经济损失也被限制在 $5 以内。

**Claude Code 的凭据安全机制：**
- API 密钥和令牌经过加密存储
- 云端执行时使用安全代理，在沙箱内使用范围限定的凭据（scoped credential），再由代理翻译为实际的 GitHub 认证令牌

### 2.5 第五层：组织管控（Managed Settings）

对于团队和组织，Claude Code 提供了**不可覆盖的管理设置（Managed Settings）**，可以通过 MDM 或 OS 级策略部署。

**管理设置中独有的安全配置：**

| 设置 | 说明 |
|------|------|
| `allowManagedHooksOnly` | 仅允许管理和 SDK Hook，阻止用户/项目 Hook |
| `allowManagedMcpServersOnly` | 仅允许管理设置中的 MCP 服务器 |
| `allowManagedPermissionRulesOnly` | 禁止用户和项目定义 allow/ask/deny 规则 |
| `sandbox.network.allowManagedDomainsOnly` | 仅允许管理设置中的域名，非白名单域名自动阻止 |
| `sandbox.filesystem.allowManagedReadPathsOnly` | 仅允许管理设置中的读取路径 |
| `disableBypassPermissionsMode` | 禁止使用 bypass 权限模式 |
| `disableAutoMode` | 禁止使用 auto 模式 |

这些设置即使用户也无法覆盖，确保组织的安全策略得到强制执行。

### 2.6 第六层：Hook 扩展的安全控制

PreToolUse Hook 提供了一种运行时权限评估机制。当 Claude Code 发起工具调用时，Hook 可以：
- **拒绝调用**（exit code 2）
- **强制提示**（要求人类确认）
- **跳过提示**（直接允许——但 deny 规则仍然生效）

一个重要的安全特性：**一个阻止型 Hook 优先于 allow 规则。** 即使某个工具已被 allow 规则允许，Hook 仍然可以在运行时拒绝它。这为组织提供了一种灵活的方式——在 allow 列表中添加 `"Bash"` 以消除所有提示，同时通过 Hook 拦截特定命令。

## 三、安全架构的互补关系

### 3.1 权限 vs 沙箱

| 维度 | 权限系统（Permissions） | 沙箱（Sandboxing） |
|------|----------------------|-------------------|
| 控制范围 | 所有工具（Bash、Read、Edit、WebFetch、MCP 等） | 仅 Bash 命令及其子进程 |
| 控制目标 | 哪些工具可用、哪些文件/域名可访问 | Shell 命令可以访问的文件系统和网络 |
| 执行层级 | Claude Code 应用层 | OS 内核层（Seatbelt/bubblewrap） |
| 对子进程生效 | 否（deny Read 规则不会阻止 `cat .env`） | 是（所有子进程继承限制） |

**关键区别：** 一个 `Read(./.env)` 的 deny 规则会阻止 Claude Code 的 Read 工具读取 `.env` 文件，但不会阻止 Bash 中的 `cat .env`。只有 OS 级沙箱能阻止所有进程访问某个路径。

### 3.2 Willison 的三层安全策略

Willison 在《Designing Agentic Loops》中总结了三种在 YOLO 模式下运行代理的策略：

**策略一：在安全沙箱中运行（推荐）**
- 使用 Docker 容器或 Apple container tool
- 限制文件和密钥访问、限制网络连接
- Willison 认为这是"reasonable risk to accept for most people"
- Anthropic 提供了官方的 Docker DevContainer 参考实现

**策略二：使用别人的计算机（Willison 最喜欢的策略）**
- GitHub Codespaces：在 Azure 虚拟机中运行，浏览器访问，有免费额度
- Code Interpreter（ChatGPT / Claude）：在隔离环境中执行代码
- OpenAI Codex Cloud：云端代理执行
- 如果代理失控，最坏的情况是代码被泄露或错误代码被推送到 GitHub 仓库——而不是你的本地机器被破坏

**策略三：直接冒险**
- 避免接触潜在的恶意指令源
- 希望能在错误造成损害之前捕获它们
- Willison 说"Most people choose option 3"

## 四、云端执行的安全模型

Claude Code 在云端执行时（Claude Code on the Web）有额外的安全控制层：

| 安全控制 | 说明 |
|----------|------|
| **隔离虚拟机** | 每个云端会话运行在独立的、Anthropic 托管的 VM 中 |
| **网络访问控制** | 默认限制网络访问，可配置为完全禁用或仅允许特定域名 |
| **凭据保护** | 通过安全代理处理认证，在沙箱内使用范围限定的凭据 |
| **分支限制** | Git push 操作限制在当前工作分支 |
| **审计日志** | 云端环境的所有操作都被记录 |
| **自动清理** | 云端环境在会话结束后自动终止 |

Remote Control 模式的安全模型不同：Web 界面连接到你本地机器上运行的 Claude Code 进程。所有代码执行和文件访问都留在本地，数据通过 TLS 加密传输。连接使用多个短期、窄范围的凭据，每个凭据仅限于特定用途并独立过期。

## 五、安全限制与已知风险

### 5.1 沙箱的安全限制

Claude Code 的官方文档明确承认了沙箱的几个已知限制：

| 限制 | 说明 |
|------|------|
| **域名前置（Domain Fronting）** | 网络过滤基于域名，但某些情况下可能通过域名前置绕过 |
| **Unix Socket 权限提升** | `allowUnixSockets` 配置可能意外授予对强大系统服务的访问权。例如，允许 `/var/run/docker.sock` 相当于授予了宿主系统的访问权限 |
| **文件系统权限提升** | 过于宽泛的文件系统写入权限可能导致权限提升。允许写入 `$PATH` 中包含可执行文件的目录、系统配置目录或用户 Shell 配置文件（`.bashrc`、`.zshrc`）是危险的 |
| **Linux 嵌套沙箱弱化** | `enableWeakerNestedSandbox` 模式允许在 Docker 环境中工作，但显著削弱了安全性 |
| **网络过滤仅基于域名** | 不过滤流量内容，用户需确保仅允许受信任的域名 |

### 5.2 内置的逃生舱口

Claude Code 包含一个有意的逃生舱口机制：当命令因沙箱限制失败时，Claude 会被提示分析失败原因，并可能使用 `dangerouslyDisableSandbox` 参数重试。使用此参数的命令会走正常的权限流程。

可以通过设置 `allowUnsandboxedCommands: false` 来禁用此逃生舱口。

### 5.3 容器逃逸

虽然 Claude Code 的沙箱使用了 OS 级别的隔离（Seatbelt/bubblewrap），但容器逃逸（Container Escape，MITRE ATT&CK T1611）始终是一个理论风险。Willison 认为这个风险是"reasonable risk to accept for most people"。

## 七、安全检查清单

基于 Claude Code 官方文档和 Willison 的建议，以下是 Agentic Engineering 实践中的安全检查清单：

**基础安全：**
- [ ] 代理是否在沙箱中运行？
- [ ] 是否限制了网络访问（仅允许必要域名）？
- [ ] 文件系统写入是否限制在工作目录内？
- [ ] 是否使用了最小权限凭据？
- [ ] 如使用凭据，是否设置了预算上限和有效期？

**提示注入防护：**
- [ ] 是否避免了让代理处理不受信任的网页内容？
- [ ] 技能文件/配置文件是否受版本控制，变更需审查？
- [ ] WebFetch 是否使用了独立的上下文窗口？
- [ ] 敏感目录（`~/.env`、密钥目录）是否在 deny 列表中？

**团队安全：**
