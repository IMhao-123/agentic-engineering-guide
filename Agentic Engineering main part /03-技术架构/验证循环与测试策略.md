## 一、验证循环：代理系统的核心机制

### 1.1 为什么验证循环如此重要

Claude Code 文档将"提供验证手段"称为**"你能做的最高杠杆的事情"**（the single highest-leverage thing you can do）：

> "Claude performs dramatically better when it can verify its own work, like run tests, compare screenshots, and validate outputs. Without clear success criteria, it might produce something that looks right but actually doesn't work. **You become the only feedback loop, and every mistake requires your attention.**"

Addy Osmani 用更简洁的话表达了同样的观点：

> "The single biggest differentiator between agentic engineering and vibe coding is testing."

### 1.2 验证循环的本质

验证循环的核心逻辑是一个"执行→检查→修正"的闭环。代理在这个循环中自主迭代，直到满足验证标准为止。

```
┌─────────────────────────────────┐
│        验证循环               │
│                                 │
│  ┌─────┐    ┌──────┐    ┌─────┐  │
│  │执行 │ → │检查  │ → │修正 │  │
│  └─────┘    └──────┘    └─────┘  │
│       ↑                        │    │
│       └────────────────────────┘    │
│                                 │
│  停止条件：                       │
│  ✓ 验证通过                    │
│  ✗ 达到迭代上限                  │
│  ✗ 需要人类判断                  │
└─────────────────────────────────┘
```

### 1.3 验证的三个层次

| 层次 | 类型 | 速度 | 可靠性 |
|------|------|------|--------|
| **自动化测试** | 单元测试、集成测试、E2E 测试 | 快 | 最高 |
| **静态分析** | 类型检查、Linting、格式化 | 快 | 高 |
| **人工审查** | 代码审查、架构审查 | 慢 | 最高 |

## 二、测试驱动开发（TDD）与代理

### 2.1 Simon Willison 的 Red/Green TDD 模式

来源：[Agentic Engineering Patterns - Red/Green TDD](https://simonwillison.net/guides/agentic-engineering-patterns/red-green-tdd/)

TDD 与编码代理的结合特别有效，原因是：**测试为代理提供了精确的、无歧义的成功标准**。

**工作流程：**

**第一步：Red（写失败测试）**

编写一个描述期望行为的测试，确保它失败。这定义了"完成"意味着什么。

**第二步：Green（让测试通过）**

给代理一个清晰目标——"让这个测试通过"。代理在循环中尝试不同的实现，直到测试变为绿色。

**第三步：Refactor（重构）**

审查并优化代码，确保测试仍然通过。

**为什么 TDD 与代理特别配合：**

| TDD 的特性 | 代理的收益 |
|-----------|-----------|
| 测试先于代码 | 代理不需要"猜测"正确的实现 |
| 明确的成功标准 | 代理知道何时停止迭代 |
| 测试即文档 | 测试本身描述了期望行为 |
| 更简洁的代码 | 代理只需编写满足测试的最少代码 |

Willison 总结："使用 TDD 模式时，代理生成的代码往往更简洁、更可靠、更少幻觉。"

### 2.2 先运行测试（First Run the Tests）

来源：[Agentic Engineering Patterns - First Run the Tests](https://simonwillison.net/guides/agentic-engineering-patterns/first-run-the-tests/)

在让代理做任何修改之前，**先运行测试确保基线是通过的**。

> "First run the tests so you know what the baseline looks like."

这为后续的代理迭代提供了一个可靠的起点。如果基线本身就有失败，代理会被混淆——它无法区分"已有的失败"和"我的修改引入的新失败"。

## 三、Claude Code 的验证机制

### 3.1 测试作为验证

Claude Code 文档给出了具体的验证实践：

| 验证类型 | 命令 | 适用场景 |
|----------|------|----------|
| 单元测试 | `pytest`、`npm test`、`go test` | 功能正确性 |
| 构建检查 | `cargo build`、`npm run build` | 编译/打包 |
| 类型检查 | `mypy`、`tsc --noEmit` | 类型安全 |
| Linting | `ruff`、`eslint` | 代码质量 |
| 端到端测试 | Playwright、Cypress | 完整的用户流程 |

Claude Code 的官方示例：

```text
claude "write tests for the auth module, run them, and fix any failures"
```

这个提示词的精妙之处在于它将三个步骤合并为一个：写测试 → 运行测试 → 修复失败。代理会在循环中完成整个流程。

### 3.2 截图作为验证

对于 UI 相关的工作，截图是一种有效的验证方式：

> "Paste screenshots directly. Copy/paste or drag and drop images into the prompt."
>
> — Claude Code Best Practices

推荐的验证流程：
1. 提供参考截图
2. 让代理实现设计
3. 让代理截取实现结果的截图
4. 让代理对比两张截图并列出差异

Claude Code 的 Chrome 扩展可以直接在浏览器中打开新标签页、测试 UI 并迭代。

### 3.3 输出验证

对于命令行工具和 API：

```text
claude -p "Implement validateEmail. Test cases: 'user@example.com' → true,
'invalid' → false. Run the tests after."
```

### 3.4 根因验证

Claude Code 文档特别强调**修复根本原因而非症状**：

| 不好的提示 | 好的提示 |
|-----------|---------|
| "the build is failing" | "the build fails with this error: [paste error]. fix it and verify the build succeeds. **address the root cause, don't suppress the error**" |

## 四、Anthropic 的评估者-优化者模式

### 4.1 在代理循环中的实现

Anthropic 的评估者-优化者模式在编码代理中自然体现为"编写→测试→修改→再测试"的循环。Claude Code 的代理循环本身就是这个模式的实例化：

1. Claude 生成代码（优化者角色）
2. Claude 运行测试（评估者角色）
3. 测试失败 → Claude 分析错误信息
4. Claude 修改代码
5. 重新运行测试（回到步骤 2）

### 4.2 编码代理作为评估者

Claude Code 的 Writer/Reviewer 模式将评估者角色提升到了独立代理层面：

- 会话 A 生成代码
- 会话 B（使用全新的上下文，避免确认偏差）审查代码
- 会话 A 根据审查反馈修复

### 4.3 评估标准

Claude Code 文档指出了两个判断模式适用性的信号：

1. **LLM 的响应能在人工反馈下改进** — 说明迭代优化有价值
2. **LLM 能自行提供有效的反馈** — 说明代理循环是可行的

## 五、CI/CD 集成

### 5.1 非交互模式

Claude Code 的非交互模式（`-p` 标志）使代理可以集成到 CI/CD 流水线中：

```bash
# 一次性查询
claude -p "Explain what this project does"

# 结构化输出供脚本解析
claude -p "List all API endpoints" --output-format json

# 流式 JSON 输出供实时处理
claude -p "Analyze this log file" --output-format stream-json
```

### 5.2 GitHub Actions 集成

Claude Code 可以作为 GitHub Actions 中的自动代码审查和 Issue 分类工具运行。Claude Code 还提供了 `--allowedTools` 标志，可以在无人值守时限制代理的能力范围。

### 5.3 GitLab CI/CD 集成

Claude Code 同样支持 GitLab CI/CD 流水线中的集成。

### 5.4 Pre-commit Hooks

Claude Code 可以通过 Hooks 机制集成到 pre-commit 工作流中。例如，配置一个 PostToolUse Hook 在每次文件编辑后自动运行 linter。

## 六、代理手动测试

### 6.1 概念

有些验证无法完全自动化——尤其是 UI 交互、用户体验、视觉布局等。Claude Code 的代理可以执行"手动"测试的自动化版本。

### 6.2 机制

| 机制 | 说明 | 工具 |
|------|------|------|
| 浏览器自动化 | 代理在真实浏览器中测试 UI | Claude in Chrome 扩展 + Playwright |
| API 测试 | 代理调用 API 端点并验证响应 | curl / httpie |
| 截图对比 | 代理截取结果并与参考图对比 | Claude in Chrome 扩展 |
| 错误追踪 | 代理记录和报告发现的问题 | Claude Code 的输出 |

### 6.3 局限性

- 代理可能遗漏只有人类能发现的交互问题
- 对视觉布局的判断不如人类精确
- 可能无法模拟所有边缘用户行为模式

## 七、质量保证清单

| 检查项 | 说明 | 状态 |
|--------|------|------|
| 代理能运行测试套件吗？ | 确保测试命令在代理环境中可用 | |
| 测试基线是通过的吗？ | 在代理修改前运行测试确认基线 | |
| 有类型检查吗？ | 配置 mypy / tsc --noEmit 并作为验证步骤 | |
| 有 Linting 吗？ | 配置 ruff / eslint 并自动运行 | |
| 有 CI 流水线吗？ | 代理生成的代码通过 CI 检查 | |
| 有人工审查吗？ | 所有代理输出经过人类审查 | |
| 有安全扫描吗？ | 依赖漏洞检查、密钥检测 | |
| 有预发布环境吗？ | 代理的修改先在预发布环境验证 | |
| 有回滚机制吗？ | Git 版本控制 + Claude Code 的 Checkpoint 系统 | |
