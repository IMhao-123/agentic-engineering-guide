## 一、为什么代理的可观测性是根本性挑战

传统软件的可观测性建立在**确定性**之上：相同的输入产生相同的输出，行为路径可预测，问题可复现。编码代理打破了这一基础——代理的决策过程是不确定的，同一个任务在不同运行中可能走完全不同的路径。这意味着：

- 你无法通过阅读代码来理解代理"为什么"做了某个决定
- 你无法通过单元测试来覆盖代理的所有行为分支
- 你无法通过传统的日志分析来追踪代理的完整决策链

Claude Code 的官方文档选择 OpenTelemetry（OTel）作为可观测性的基础，这一选择本身就说明了问题的规模：OpenTelemetry 原本是用于分布式系统追踪的工业标准，而编码代理的可观测性需求已经达到了分布式系统的复杂度级别。

## 二、Claude Code 的 OpenTelemetry 集成

### 2.1 三种遥测信号

Claude Code 通过 OpenTelemetry 导出三种遥测信号：

| 信号 | 协议 | 说明 |
|------|------|------|
| **Metrics（指标）** | OTLP / Prometheus / Console | 时间序列数据，用于趋势分析和告警 |
| **Logs/Events（事件）** | OTLP / Console | 离散事件，用于审计和分析 |
| **Traces（追踪，Beta）** | OTLP / Console | 分布式追踪，将单个用户提示关联到所有触发的 API 请求和工具执行 |

**启用方式（最简配置）：**

```bash
# 启用遥测
export CLAUDE_CODE_ENABLE_TELEMETRY=1

# 选择指标导出器
export OTEL_METRICS_EXPORTER=otlp

# 选择事件导出器
export OTEL_LOGS_EXPORTER=otlp

# 配置 OTLP 端点
export OTEL_EXPORTER_OTLP_PROTOCOL=grpc
export OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4317

# 启用追踪（Beta）
export CLAUDE_CODE_ENHANCED_TELEMETRY_BETA=1
export OTEL_TRACES_EXPORTER=otlp

# 运行 Claude Code
claude
```

**组织级部署：** 管理员可以通过管理设置（Managed Settings）集中配置遥测，用户无法覆盖。这使得组织可以为所有开发者统一可观测性策略。

### 2.2 指标（Metrics）

Claude Code 导出以下指标：

| 指标名 | 说明 | 单位 | 额外属性 |
|--------|------|------|----------|
| `claude_code.session.count` | 启动的 CLI 会话数 | count | — |
| `claude_code.lines_of_code.count` | 修改的代码行数 | count | `type`: "added" / "removed" |
| `claude_code.pull_request.count` | 创建的 PR 数 | count | — |
| `claude_code.commit.count` | 创建的 Git 提交数 | count | — |
| `claude_code.cost.usage` | 会话成本 | USD | `model` |
| `claude_code.token.usage` | Token 使用量 | tokens | `type`: "input"/"output"/"cacheRead"/"cacheCreation", `model` |
| `claude_code.code_edit_tool.decision` | 代码编辑工具权限决策数 | count | `tool_name`, `decision` ("accept"/"reject"), `source`, `language` |
| `claude_code.active_time.total` | 总活跃时间 | seconds | `type`: "user"（键盘交互）/ "cli"（工具执行和 AI 响应） |

**所有指标共享的标准属性：**
- `session.id`：唯一会话标识符（可通过 `OTEL_METRICS_INCLUDE_SESSION_ID` 控制）
- `app.version`：Claude Code 版本号
- `organization.id`：组织 UUID
- `user.account_uuid` / `user.account_id`：账户标识
- `user.email`：用户邮箱（OAuth 认证时）
- `terminal.type`：终端类型（iTerm.app、vscode、cursor、tmux 等）

**基数控制：** 为防止指标存储爆炸，可通过环境变量控制属性的包含：
- `OTEL_METRICS_INCLUDE_SESSION_ID`（默认 true）
- `OTEL_METRICS_INCLUDE_VERSION`（默认 false）
- `OTEL_METRICS_INCLUDE_ACCOUNT_UUID`（默认 true）

### 2.3 事件（Events）

事件是离散的、带时间戳的记录，用于细粒度分析和审计。

**事件关联机制——prompt.id：**

每个事件都包含 `prompt.id` 属性（UUID v4），将所有在处理单个用户提示期间产生的事件关联在一起。这构成了事件关联的核心：通过一个 `prompt.id`，可以追踪到用户提示事件、所有 API 请求事件、以及所有工具执行结果事件。

| 事件名 | 触发时机 | 关键属性 |
|--------|----------|----------|
| `claude_code.user_prompt` | 用户提交提示时 | `prompt_length`, `prompt`（默认脱敏） |
| `claude_code.tool_result` | 工具完成执行时 | `tool_name`, `success`, `duration_ms`, `error`, `decision_type`, `decision_source` |
| `claude_code.api_request` | 每次 API 请求 | `model`, `cost_usd`, `duration_ms`, `input_tokens`, `output_tokens`, `cache_read_tokens`, `cache_creation_tokens`, `speed` |
| `claude_code.api_error` | API 请求失败时 | `model`, `error`, `status_code`, `duration_ms`, `attempt`, `speed` |
| `claude_code.tool_decision` | 工具权限决策时 | `tool_name`, `decision`, `source` |

**`prompt.id` 被有意排除在指标之外**，因为每个提示都生成唯一的 ID，会创建无限增长的时间序列。它仅用于事件级别的分析和审计追踪。

**详细日志控制：**

| 环境变量 | 默认 | 说明 |
|----------|------|------|
| `OTEL_LOG_USER_PROMPTS` | 禁用 | 记录用户提示内容 |
| `OTEL_LOG_TOOL_DETAILS` | 禁用 | 记录工具参数（Bash 命令、MCP 服务器/工具名、技能名） |
| `OTEL_LOG_TOOL_CONTENT` | 禁用 | 记录工具输入和输出内容（限 60KB，需启用追踪） |

这些详细日志默认禁用，因为可能包含敏感信息。Claude Code 官方文档建议配置遥测后端来过滤或脱敏这些属性。

### 2.4 分布式追踪（Traces, Beta）

追踪是可观测性中最强大的信号，它将每个用户提示关联为一个完整的**追踪（Trace）**，其中包含多个**跨度（Span）**——每个 API 请求和工具执行都是一个 Span。

**启用配置：**
```bash
export CLAUDE_CODE_ENHANCED_TELEMETRY_BETA=1
export OTEL_TRACES_EXPORTER=otlp
```

追踪与指标/事件的关键区别：
- 指标告诉你"发生了多少次"
- 事件告诉你"什么时候发生了什么"
- 追踪告诉你"这一系列操作是如何串联的"

对于调试代理行为和理解决策链路，追踪是最有价值的信号。

### 2.5 后端选择

Claude Code 的官方文档根据不同需求推荐了不同的后端：

| 后端类型 | 适用信号 | 代表产品 | 优势 |
|----------|----------|----------|------|
| 时序数据库 | Metrics | Prometheus | 速率计算、聚合指标 |
| 列式存储 | Metrics + Events | ClickHouse | 复杂查询、唯一用户分析 |
| 全功能可观测平台 | 所有三种 | Honeycomb, Datadog | 高级查询、可视化、告警 |
| 日志聚合 | Events | Elasticsearch, Loki | 全文搜索、日志分析 |
| 分布式追踪 | Traces | Jaeger, Zipkin, Grafana Tempo | Span 可视化、请求瀑布图、延迟分析 |

## 三、多团队组织支持

### 3.1 团队维度的资源属性

组织可以通过 `OTEL_RESOURCE_ATTRIBUTES` 为遥测数据添加自定义属性，以区分不同的团队或部门：

```bash
export OTEL_RESOURCE_ATTRIBUTES="department=engineering,team.id=platform,cost_center=eng-123"
```

这使得组织可以：
- 按团队或部门过滤指标
- 按成本中心追踪成本
- 创建团队特定的仪表盘
- 为特定团队设置告警

### 3.2 动态认证头

对于需要动态认证的企业环境（如短期令牌），Claude Code 支持通过脚本动态生成认证头：

```json
{
  "otelHeadersHelper": "/bin/generate_opentelemetry_headers.sh"
}
```

脚本输出 JSON 格式的 HTTP 头，默认每 29 分钟刷新一次。

## 四、评估（Evaluation）：比测试更难的挑战

### 4.1 评估 ≠ 测试

- **测试**验证代码是否满足功能规格——输入确定，输出确定
- **评估**衡量代理在不确定性中的整体表现——输入确定，输出不确定

这个区别是根本性的。传统的单元测试、集成测试都可以用来验证代理生成的代码质量，但它们无法回答一个更基本的问题：**代理到底做得怎么样？**

### 4.2 学术视角：Agentic AI 评估的可重现性危机

来源：arXiv:2604.01437《Reproducible, Explainable, and Effective Evaluations of Agentic AI for Software Engineering》（Li & Storhaug, FSE 2026 ResponsibleSE 研讨会）

这篇论文系统分析了 ICSE 2025-2026、FSE 2025、ASE 2025、ISSTA 2025 中的 18 篇 Agentic AI for SE 论文，揭示了当前评估实践的严重不足：

**核心发现：**

1. **评估设计描述中的信息缺失使得结果无法重现**——这是最严重的问题。许多论文没有提供足够的细节让其他研究者复现实验。
2. **LLM 本质上是黑盒**——很难证明 Agentic AI 方法优于基线方法，因为无法解释差异的来源。
3. **缺少标准化的评估框架**——每个研究使用不同的基准和指标，难以跨研究比较。

**论文的核心建议：公开 Thought-Action-Result（TAR）轨迹和 LLM 交互数据。**

TAR 轨迹的格式：
```
[T] Thought: 我需要先理解这个函数的作用
[A] Action: read_file("src/utils.py")
[R] Result: 函数 parse_config() 解析 JSON 配置文件...
[T] Thought: 这个函数缺少错误处理
[A] Action: edit_file("src/utils.py", ...)
[R] Result: 已添加 try-except 块
```

论文认为，公开 TAR 轨迹（或其摘要版本）能让后续研究系统地分析不同方法的优缺点。他们提供了一个概念验证案例研究，展示了如何利用 TAR 轨迹进行跨方法的系统性比较。

### 4.3 评估维度

| 维度 | 说明 | 衡量指标 |
|------|------|----------|
| **准确度（Accuracy）** | 代理完成任务的程度 | 任务完成率、SWE-bench 分数 |
| **可靠性（Reliability）** | 代理在多次运行中的一致性 | 成功率的标准差 |
| **性能（Performance）** | 代理的速度和效率 | 执行时间、Token 消耗、API 成本 |
| **安全性（Safety）** | 代理的安全性表现 | 攻击抵抗率（ClawSafety ASR）、数据泄露事件 |

### 4.4 Claude Code 遥测数据支持的评估

通过 Claude Code 的 OpenTelemetry 集成，可以实际构建以下评估：

**使用模式分析：**
- `claude_code.token.usage` 按 `type`（input/output/cacheRead/cacheCreation）分解——分析缓存命中率
- `claude_code.session.count` 按时间追踪——分析采用趋势
- `claude_code.lines_of_code.count` 按 `type`（added/removed）分解——衡量生产力
- `claude_code.commit.count` + `claude_code.pull_request.count`——衡量对开发流程的影响

**成本分析：**
- `claude_code.cost.usage` 按 `model` 分解——比较不同模型的成本效益
- 按 `user.account_uuid` 分解——识别高使用量的用户

**工具使用模式：**
- 分析 `tool_result` 事件——最常用的工具、工具成功率、平均执行时间、错误模式

**权限决策分析：**
- `claude_code.code_edit_tool.decision` 按 `decision`（accept/reject）和 `source`（config/hook/user_permanent/user_temporary/user_abort/user_reject）分解——理解用户与代理的交互模式

## 五、Agno AgentOS 的控制面板

Agno 的 AgentOS 提供了一个 Web 控制面板（Control Plane），从三个维度评估代理：

| 维度 | 说明 |
|------|------|
| **准确度（Accuracy）** | 代理回答的正确性 |
| **可靠性（Reliability）** | 代理在不同情况下的一致表现 |
| **性能（Performance）** | 代理的响应速度和资源消耗 |

控制面板还提供：
- **会话监控：** 每个实时交互的深入洞察
- **记忆管理器：** 查看、编辑和组织代理的记忆
- **知识管理器：** 管理代理使用的知识库

Agno 在 2026 年 3 月发布了原生追踪功能（Native Tracing），其设计理念是"完整的代理可观测性不应该需要安全审查"——追踪集成在 AgentOS 内部，不需要将敏感数据发送给第三方。

## 六、调试代理的独特挑战

调试编码代理比调试传统程序困难得多，原因在于：

**不确定性：** 同样的输入可能产生不同的输出。代理可能通过非预期的路径达成目标。同一个 Bug 在第一次运行中可能被修复，在第二次运行中可能以完全不同的方式被处理。

**问题归因困难：** 问题可能出在多个层面：
- LLM 的推理能力不足
- 上下文信息缺失或过载
- 工具使用方式不当
- 权限设置过于严格或过于宽松
- 提示词设计不合理

**Claude Code 提供的调试支持：**

| 方法 | 说明 |
|------|------|
| **追踪回放** | 通过分布式追踪回放代理的完整执行轨迹，理解决策链路 |
| **检查点恢复** | 在代理修改代码之前创建检查点，出问题时恢复 |
| **会话分支** | 从某个节点分支出一个新的会话，尝试不同的方法 |
| **模型降级** | 使用更简单/更确定的模型排除推理问题 |
| **权限审计** | 通过 `/permissions` 查看所有权限规则和来源 |
| **遥测分析** | 通过 OpenTelemetry 数据分析代理的使用模式和成本 |

## 七、ROI 测量

Claude Code 的 ROI 测量指南（`github.com/anthropics/claude-code-monitoring-guide`）提供了一个完整的框架，包括：
- 遥测设置
- 成本分析
- 生产力指标

这个指南包含可用的 Docker Compose 配置、Prometheus 和 OpenTelemetry 设置，以及生产力报告模板。
