## 一、状态持久化的三个层次

代理系统的持久化需求覆盖从单次会话到跨会话的知识积累。

### 1.1 层级总览

| 层次 | 持久化对象 | 持久化方式 | 生命周期 |
|------|-----------|-----------|----------|
| **会话级** | 对话历史、任务状态 | 本地文件 | 单次会话 |
| **项目级** | 项目模式、学习经验 | CLAUDE.md + Auto Memory | 跨会话 |
| **代理级** | 代理的领域知识积累 | 子代理 Memory | 跨项目 |

## 二、会话级持久化

### 2.1 Claude Code 的会话管理

Claude Code 自动将所有对话持久化到本地存储。每次消息、工具调用和结果都被记录，这使得：

- **回滚（Rewind）** 成为可能——可以恢复对话和/或代码到之前的状态
- **恢复（Resume）** 成为可能——关闭终端后可以继续之前的对话
- **分支（Fork）** 成为可能——从某个对话点创建新的分支

### 2.2 恢复会话

```bash
claude --continue    # 继续最近的对话
claude --resume      # 从历史会话中选择
```

恢复会话时，完整的对话历史被恢复，但会话级的权限设置不会被继承（需要重新批准）。

### 2.3 分支会话

```bash
claude --continue --fork-session
```

这会从当前会话的对话历史创建一个新的会话 ID，保留到该点之前的所有上下文。原始会话不受影响。

**典型用途：** 当代理在一个方向上走了很远但发现不对时，可以分支出来尝试不同的方法，而不影响原始的对话路径。

### 2.4 会话命名

```bash
claude --resume  # 选择时给会话命名
```

Claude Code 支持用描述性名称管理会话（如 "oauth-migration"、"debugging-memory-leak"），使其像 Git 分支一样易于管理。

### 2.5 跨终端共享的注意事项

如果多个终端恢复同一个会话，两个终端的消息会交错写入——就像两个人在同一个笔记本上写字。虽然不会损坏数据，但对话会变得混乱。

对于需要从同一起点并行工作的场景，应使用 `--fork-session` 为每个终端创建独立的会话。

## 三、代码级持久化：Checkpoints

### 3.1 工作原理

Claude Code 在每次文件编辑前**自动创建 Checkpoint**——快照当前文件内容。

这意味着：
- 文件的每一次修改都是**完全可逆**的
- 即使代理进行了 50 步修改，也可以回滚到任何一步
- Checkpoint 是本地到会话的，不替代 Git

### 3.2 回滚操作

| 操作 | 方法 | 效果 |
|------|------|------|
| 轻度回滚 | 双击 `Esc` 打开回滚菜单 | 恢复对话/代码/或两者到之前的状态 |
| 部分回滚 | 选择某个消息点后选择"Summarize from here" | 总结该点之后的内容，保留之前的上下文 |
| 完全重置 | `/clear` | 清除所有对话历史 |

### 3.3 Checkpoint 的局限性

Claude Code 文档明确警告：

> "Checkpoints only track changes made *by Claude*, not external processes. This isn't a replacement for git."

Checkpoint 不覆盖：
- 外部进程对文件的修改（如你手动编辑了文件）
- 远程系统的变更（数据库修改、API 调用、部署操作）

这些操作不受 Checkpoint 保护，这就是为什么 Claude Code 仍然会询问确认涉及外部副作用的命令执行。

### 3.4 跨会话的 Checkpoint

Checkpoints 持久存在会话文件中。即使关闭终端后重新打开 Claude Code，仍然可以回滚到之前的状态。

## 四、项目级持久化：CLAUDE.md 与 Auto Memory

### 4.1 CLAUDE.md：人工维护的持久化知识

CLAUDE.md 是开发者为代理编写的项目级指令文件。它通过版本控制（提交到 Git）实现持久化和团队共享。

**更新方式：** 手动编写和维护。

**更新时机：** 当发现代理反复犯同一错误时，说明 CLAUDE.md 需要添加或修改规则。

**维护原则：**

> "Treat CLAUDE.md like code: review it when things go wrong, prune it regularly, and test changes by observing whether Claude's behavior actually shifts."
>
> — Claude Code 文档

### 4.2 Auto Memory：代理自动积累的知识

Auto Memory 是 Claude Code 自动维护的持久化学习记录。它存储在项目根目录的 `.claude/` 目录下。

**工作方式：**

1. Claude 在工作中发现有用的模式（如构建命令、调试经验）时，自动保存到 MEMORY.md
2. 每次会话开始时，Auto Memory 的前 200 行或 25KB（取较小者）自动加载到上下文中
3. 如果 MEMORY.md 超过限制，代理被指示精简它

**内容类型：**
- 项目模式（如"payment 模块使用事件驱动架构"）
- 构建命令和调试经验
- 用户的个人偏好

### 4.3 CLAUDE.md 与 Auto Memory 的互补

| 维度 | CLAUDE.md | Auto Memory |
|------|-----------|-------------|
| 维护者 | 人工 | 代理自动 |
| 内容类型 | 规则和约束 | 经验和学习 |
| 稳定性 | 稳定（版本控制） | 不断积累 |
| 适用场景 | 代理"必须遵守"的事情 | 代理"已经学到"的事情 |

## 五、代理级持久化：子代理 Memory

### 5.1 概念

子代理可以通过 `memory` 字段拥有自己的跨会话持久化记忆目录。

### 5.2 记忆范围选择

| 范围 | 存储位置 | 适用场景 |
|------|----------|----------|
| `user` | `~/.claude/agent-memory/<name>/` | 跨所有项目的通用知识 |
| `project` | `.claude/agent-memory/<name>/` | 项目特定知识，可通过 Git 共享 |
| `local` | `.claude/agent-memory-local/<name>/` | 项目特定但不提交到 Git |

推荐使用 `project` 范围——它使知识可以通过版本控制共享，且不会意外暴露本地特定的信息。

### 5.3 记忆的工作方式

当启用 memory 时，子代理的系统提示会包含读写记忆目录的指令，且 MEMORY.md 的前 200 行会在每次会话开始时加载。

**推荐的提示词模式：**

```text
Review this PR, and check your memory for patterns you've seen before.

Now that you're done, save what you learned to your memory.
```

第一句：让代理查阅已有的记忆，避免重复犯错。
第二句：让代理在任务完成后更新记忆，积累新知识。

### 5.4 记忆积累的效果

随着会话次数增加，子代理的记忆会变得越来越有价值：
- 代码库中的常见模式和约定
- 反复出现的 Bug 类型及其修复方法
- 项目特定的架构决策
- 特定领域的最佳实践

## 六、运行时级持久化：数据库集成

### 6.1 Agno 的数据库集成

Agno Framework 原生支持将代理状态持久化到数据库：

```python
agent = Agent(
    name="Knowledge Agent",
    model="claude:sonnet-4",
    db=Postgres(connection_string),  # 所有状态持久化到数据库
)
```

Agno 支持的数据库类型：
- PostgreSQL
- SQLite
- 其他 SQLAlchemy 支持的数据库

**持久化的内容包括：**
- 会话历史
- 代理记忆
- 知识库条目
- 工作流状态
- 审计日志

### 6.2 OpenAI Agents SDK 的 Redis 支持

OpenAI Agents SDK 支持 Redis 作为会话存储后端：

```bash
pip install 'openai-agents[redis]'
```

Redis 提供了：
- 高性能的会话读写
- 分布式环境中的共享会话
- 会话的 TTL（Time to Live）管理

### 6.3 数据库 vs 文件系统的选择

| 维度 | 文件系统 | 数据库 |
|------|----------|--------|
| 适用规模 | 小型项目、个人使用 | 中大型系统、团队协作 |
| 查询能力 | 无（只能遍历文件） | 强（SQL 查询） |
| 并发支持 | 弱 | 强 |
| 运维复杂度 | 低 | 高 |
| 适合代理 | CLAUDE.md + Auto Memory | Agno AgentOS 级别的完整持久化 |

## 七、容错机制

### 7.1 失败检测

| 失败类型 | 检测方式 |
|----------|----------|
| 工具调用失败 | 工具返回的错误 |
| 测试失败 | 测试运行器的退出码 |
| 构建失败 | 构建命令的输出 |
| 上下文溢出 | 自动压缩触发或手动 /context 检查 |
| 无限循环 | maxTurns 配置项 |
| 权限不足 | 权限提示或分类器阻止 |

### 7.2 恢复策略

| 策略 | 说明 | 实现方式 |
|------|------|----------|
| **重试** | 自动重试可恢复的失败 | 代理循环的自然行为 |
| **回滚** | 恢复到之前的状态 | Claude Code 的 Checkpoint + `/rewind` |
| **清除重试** | 清除污染的上下文后重新开始 | `/clear` |
| **降级** | 切换到更简单的方法 | 代理自行决策或人工干预 |
| **求助** | 暂停并请求人类帮助 | AskUserQuestion 工具 |
| **终止** | 在无法恢复时优雅终止 | 自动模式回退或最大迭代次数 |

### 7.3 幂等性设计

确保代理的操作是幂等的（多次执行结果相同）可以大幅降低容错的复杂度：

- 数据库操作使用 `INSERT ... ON CONFLICT` 代替 `INSERT`
- 文件写入使用原子操作
- 避免有副作用的操作在检查点之前执行
- 使用声明式配置代替命令式操作

### 7.4 子代理的容错

子代理的容错与主对话略有不同：

| 场景 | 行为 |
|------|------|
| 子代理因缺少权限而失败 | 以前台模式启动新的前台子代理重试（此时会有交互式权限提示） |
| 子代理上下文溢出 | 子代理的自动压缩功能会自动触发（约 95% 时） |
| 子代理执行超时 | 可配置 `maxTurns` 限制最大迭代次数 |
| 子代理不可恢复的错误 | Claude Code 将错误信息返回给主对话，由主对话决定下一步 |

## 八、会话迁移

### 8.1 跨平台迁移

Claude Code 支持在多个界面间迁移会话：

| 从 | 到 | 方法 |
|----|---|------|
| 终端 | 桌面应用 | `/desktop` 命令 |
| 终端 | Web | `claude --teleport` |
| 终端 | 手机 App | 远程控制（Remote Control） |
| Web | 终端 | `claude --teleport` |
| 手机 App | 桌面应用 | Dispatch |

### 8.2 分支迁移

`--fork-session` 标志可以从某个对话点创建新的分支。这比 Git 的 fork 更轻量——它只克隆对话历史，不涉及代码文件。

### 8.3 跨目录迁移

Claude Code 的会话绑定到当前目录。当你恢复会话时，只能看到该目录下的会话。切换 Git 分支后，会话历史保持不变，但代理可以看到新分支的文件。
