> **核心来源：**
> - Simon Willison《Anti-patterns: things to avoid》
> - Addy Osmani《Agentic Engineering》及《Stop Using /init for AGENTS.md》
> - Anthropic《Building Effective Agents》
> - ClawSafety 论文（arXiv:2604.01438）
> - arXiv:2604.01532《PHMForge》

## 一、来自一手资料的反模式

### 1.1 头号反模式：将未经审查的代码强加给协作者

来源：Simon Willison《Anti-patterns》

> "Don't file pull requests with code you haven't reviewed yourself."

这是 Willison 明确列出的最严重的反模式。

**为什么这是反模式：**
- 破坏团队信任——你的同事花了时间审查你本该自己审查的代码
- 在代码库中引入未知的复杂度
- 使其他团队成员难以理解和维护代码
- 代理生成的代码可能"看起来能用"但隐藏了深层问题
- 代理会生成"令人信服但可能不准确"的 PR 描述

**正确的 PR 实践：**
- 代码能工作，你确信它能工作
- 变更足够小以被高效审查
- 包含额外上下文（关联 issue、规格说明）
- 包含人工验证的证据（测试笔记、截图、实现选择的说明）

### 1.2 将 Agentic Engineering 与 Vibe Coding 混为一谈

来源：Addy Osmani《Agentic Engineering》

> "The problem is that 'vibe coding' has become a suitcase term. People now use it to describe everything from a weekend hack to a disciplined engineering workflow where AI agents handle implementation under human oversight. These are fundamentally different activities, and conflating them is causing real confusion—and real damage."

**区分标准：** Agentic Engineering 有规格说明、有测试、有审查、有工程纪律。Vibe Coding 没有。

### 1.3 过早引入框架复杂度

来源：Anthropic《Building Effective Agents》

> "They often create extra layers of abstraction that can obscure the underlying prompts and responses, making them harder to debug. They can also make it tempting to add complexity when a simpler setup would suffice."

Anthropic 建议从直接使用 LLM API 开始，只在确实需要时才引入框架。

### 1.4 自动生成泛化的 AGENTS.md

来源：Addy Osmani《Stop Using /init for AGENTS.md》

> "Stop running /init. The auto-generated output is redundant with your existing documentation and adds overhead without benefit."

两篇学术论文（Lulla et al. ICSE JAWs 2026、ETH Zurich 2026）证实：LLM 自动生成的上下文文件使任务成功率降低 2-3%，成本增加 20%+。原因：生成的内容与代理自己能发现的信息重复，增加了噪音而非价值。

**正确做法：** 手工编写，只包含代理无法通过阅读代码发现的信息。

### 1.5 过早跳到多代理系统

来源：Agno《The 5 levels of agentic software》

> "Most teams skip straight to this level. That's the mistake the framework is specifically designed to prevent."

> "A single agent with good instructions and proper infrastructure would have solved 80% of their problem without the coordination failures, context confusion, or spiraling costs."

## 二、常见的实践陷阱

### 2.1 没有测试就启动代理循环

**问题：** 没有测试，代理无法知道自己的修改是否正确。代理会"愉快地宣称完成"但实际上代码是坏的。

**后果：** 代理在错误方向上浪费大量 Token 和时间。

**解决：** 先建立测试套件，再启动代理循环。如果还没有测试，先让代理帮你建立——然后在一个新的会话中用 TDD 模式开始实际工作。

### 2.2 目标不明确

**问题：** 给代理一个模糊的目标（如"改进代码质量"），代理无法知道何时停止。

**后果：** 代理可能产生大量无用的修改，或者在一个方向上无限循环。

**解决：** 给代理一个具体的、可验证的目标。"让 `test_user_registration` 通过"比"改进认证模块"好一万倍。

### 2.3 在本地直接运行 YOLO 模式

**问题：** 不在沙箱中运行代理，允许代理无限制地执行命令。

**后果：**
- 删除重要文件
- 窃取密钥和凭据（环境变量中的 API keys）
- 意外的网络操作（Willison 警告的三大 YOLO 风险之一）
- ClawSafety 研究显示攻击成功率在 40%-75%

**解决：**
- 至少在 Docker 容器中运行
- Anthropic 官方推荐：在无网络的容器中运行 `--dangerously-skip-permissions`
- Willison 推荐的首选方案：使用 GitHub Codespaces 或 Codex Cloud（在别人的机器上运行）

### 2.4 忽视成本

**问题：** 代理循环会消耗大量 Token，尤其是使用强能力模型时。

**后果：** API 账单可能远超预期。

**解决：**
- 设置 Token 预算
- 根据任务复杂度选择模型级别（不需要 Opus 的任务用 Sonnet）
- 利用 Prompt Caching 减少重复计算
- 通过 OpenTelemetry 监控 `claude_code.cost.usage` 和 `claude_code.token.usage`

### 2.5 技能退化

来源：Addy Osmani

> "If you're junior and you lean on AI before building those fundamentals, you risk a dangerous skill atrophy. You can produce code without understanding it. You can ship features without learning why certain patterns exist."

**问题：** 初学者过度依赖 AI，不建立基础知识。

**后果：** 能生成代码但不能调试，能提交功能但不能推理。当 AI 无法解决问题时，没有能力兜底。

**解决：**
- 先建立基础知识，再使用 AI 加速
- 用 AI 辅助学习，而非替代学习
- 审查和理解代理生成的每一行代码

### 2.6 代理间的代码冲突

**问题：** 多个代理同时修改代码库的相同文件或相关文件。

**后果：** Git 合并冲突、代码不一致、逻辑错误。

**解决：**
- 使用 Git 分支或 worktree 隔离各代理的工作
- 确保各代理操作的文件不重叠
- 使用子代理或代理团队模式而非完全并行的独立实例
- Claude Code 官方推荐使用 Git Worktree 运行并行会话

### 2.7 盲目信任代理的安全声明

来源：ClawSafety 研究（arXiv:2604.01438）

> "Safety is not determined by the backbone model alone but depends on the full deployment stack."

**问题：** 认为使用"安全的"模型或框架就足够保护系统。

**后果：** 即使最强的模型也有 40% 的攻击成功率；同一模型在不同框架下安全表现差异显著。

**解决：**
- 采用纵深防御（权限 + 沙箱 + Hook + 管理设置）
- 不信任任何单一安全层
- 定期通过 OpenTelemetry 监控和审计代理行为

### 2.8 AGENTS.md 中的锚定效应

来源：Osmani《Stop Using /init for AGENTS.md》

**问题：** 在 AGENTS.md 中提到某个技术（如 tRPC），即使它只是少数遗留端点在使用，代理也会在所有任务中倾向使用它。

**后果：** 代理被上下文中的过时信息误导，生成不符合当前代码库实际的代码。

**解决：**
- 只写代理无法自己发现的信息
- 定期审查和更新 AGENTS.md
- 考虑使用动态加载的分层架构替代单一静态文件

## 三、认知陷阱

### 3.1 "AI 能处理一切" 的幻觉

代理能处理的事情比大多数人想象的少，比 AI 营销宣传的更少。

PHMForge 研究（arXiv:2604.01532）的数据：即使最高性能配置（Claude Sonnet 4.0 + ReAct + Claude Code），任务完成率也仅为 68%。这意味着近三分之一的任务，最强的代理配置也无法独立完成。

系统性的失败模式：工具编排错误（23% 的任务是错误排序的）、多资产推理下降（14.9 百分点）。

### 3.2 "更快就是更好" 的错觉

Agentic Engineering 不一定更快——它用编码时间换来了审查时间和编排技能。Willison 承认："AI-generated code needs to be reviewed, which means the natural bottleneck on all of this is how fast I can review the results."

真正的优势在于产出质量，而非速度。代理使你能够同时推进多个任务、快速尝试多种方案、消除过去"太耗时"的质量改进——但这些都需要你投入时间审查和验证。

### 3.3 "代理生成的代码不需要审查" 的危险信念

这是 Vibe Coding 的核心谬误。Willison 的头号反模式就是专门针对这个问题的。无论代码是谁生成的，都需要经过同等严格的审查。

### 3.4 "我的直觉知道什么对代理有用" 的过度自信

Osmani 引用的 Arize AI 研究表明：帮助人类理解代码库的信息和帮助 LLM 导航代码库的信息往往是不同的。你以为"显然有用"的指令对模型可能是噪音，模型实际需要的信息你可能从未想过要写下来。

## 四、反模式严重程度排序

| 反模式 | 严重程度 | 正确做法 |
|--------|----------|----------|
| 未经审查的代码合入 | 极高 | 以人类 PR 标准审查代理代码 |
| 无测试启动代理循环 | 高 | 先建立测试套件 |
| YOLO 模式无沙箱 | 高 | 至少在 Docker 容器中运行 |
| 模糊目标 | 高 | 给代理具体可验证的目标 |
| 与 Vibe Coding 混淆 | 高 | 理解并坚持工程纪律 |
| 盲目信任安全声明 | 高 | 纵深防御 |
| 过早引入框架复杂度 | 中 | 从直接 API 调用开始 |
| 过早跳到多代理 | 中 | 先证明单代理不够用 |
| 忽视成本 | 中 | 设置 Token 预算并监控 |
| 技能退化 | 中 | 建立基础后再使用 AI |
| 自动生成泛化配置 | 中 | 手工编写有针对性的配置 |
| AGENTS.md 锚定效应 | 低-中 | 只写不可发现的信息，定期更新 |
| 代理间代码冲突 | 低-中 | Git Worktree 隔离 |
| 对代理能力过度自信 | 低 | 记住 68% 的上限 |
