## 一、什么是代理循环

Willison 的定义：

> "My preferred definition of an LLM agent is something that runs tools in a loop to achieve a goal. The art of using them well is to carefully design the tools and loop for them to use."
>
> — Simon Willison《Designing Agentic Loops》

代理循环的本质是：**将问题简化为一个清晰的目标 + 一组可以在循环中验证和迭代的工具。** 如果你能做到这一点，编码代理往往能通过蛮力找到有效方案。

## 二、设计流程

### 第一步：识别适合代理循环的任务

适合的问题具有以下特征：

| 特征 | 说明 | Willison 的表述 |
|------|------|----------------|
| **明确的目标** | 可以清晰定义"完成"意味着什么 | Clear success criteria |
| **可测试的结果** | 有客观方式判断是否成功 | Objective way to judge success |
| **试错空间** | 需要多次尝试才能找到好的方案 | Trial and error involved |
| **工具可用** | 有工具可以在循环中验证和迭代 | Tools available to iterate |

**识别信号：**

> "Any time you find yourself thinking 'ugh, I'm going to have to try a lot of variations here' is a strong signal that an agentic loop might be worth trying!"
>
> — Simon Willison

每当你的直觉说"呃，我要尝试很多变体了"——这就是使用代理循环的信号。

### 第二步：定义目标

**好的目标——具体且可验证：**
- "让 `test_user_registration` 测试通过"
- "将 Docker 镜像大小减少到 200MB 以下"
- "修复 CI 中所有 flaky test"

**不好的目标——模糊且主观：**
- "改进代码质量"
- "让应用更快"
- "重构这个模块"

代理需要一个可以客观判断"成功"的标准。测试是最常见也最可靠的成功标准。

### 第三步：准备工具

| 工具类型 | 示例 | 用途 |
|----------|------|------|
| 测试运行器 | `pytest`、`npm test`、`go test` | 验证代码正确性 |
| 文件操作 | 读取、编写、编辑文件 | 修改代码 |
| Shell 命令 | 构建、部署、安装依赖 | 执行操作 |
| 搜索工具 | `grep`、`rg`、Glob | 导航代码库 |
| Linting | `ruff`、`eslint`、`mypy` | 代码质量检查 |
| Git | `git log`、`git bisect` | 追溯变更历史 |

Willison 建议将常用工具的使用方法写入 AGENTS.md：

> "I find it's usually more productive to think in terms of shell commands instead. Coding agents are really good at running shell commands!"
>
> — Simon Willison

不需要事无巨细——给一个例子，代理就能推断其他用法的变体。

### 第四步：设计循环

```
┌─────────────────────────────────────┐
│           代理循环                    │
│                                     │
│  ┌──────┐    ┌──────┐    ┌──────┐  │
│  │ 思考 │ →  │ 行动 │ →  │ 验证 │  │
│  └──────┘    └──────┘    └──────┘  │
│       ↑                        │    │
│       └────────────────────────┘    │
│                                     │
│  停止条件：                          │
│  - 目标达成（测试通过）               │
│  - 达到迭代上限                      │
│  - 遇到无法恢复的错误                 │
│  - 需要人类决策                      │
└─────────────────────────────────────┘
```

Claude Code 的三阶段循环模型（来自官方文档）：
1. **Gather Context（收集上下文）：** 读取相关文件、搜索代码库
2. **Take Action（执行行动）：** 编辑文件、运行命令
3. **Verify Results（验证结果）：** 运行测试、检查输出

### 第五步：设置安全边界

- 确保代理在沙箱中运行（Docker 容器 + 网络白名单）
- 设置凭据和预算限制
- 配置最大迭代次数或 Token 预算
- 设置停止条件

## 三、实战案例

### 案例一：调试失败的测试

**场景：** `test_order_total` 测试失败

**目标：** 让 `test_order_total` 测试通过

**Willison 的观察：**
> "A test is failing and you need to investigate the root cause. Coding agents that can already run your tests can likely do this without any extra setup."

调试是代理循环最天然的应用场景——代理可以运行测试、分析失败信息、定位问题代码、提出修复方案、验证修复——整个循环的每一步都有工具支持。

### 案例二：性能优化

**场景：** 一个 SQL 查询太慢

**目标：** 将查询响应时间降低到 100ms 以下

**工具准备：**
- 数据库 CLI（运行和基准测试查询）
- 文件读写工具（修改 ORM 查询）
- `EXPLAIN ANALYZE`（查看查询计划）

代理循环：运行基准 → 分析查询计划 → 尝试优化 → 验证改进且功能正确 → 如果未达标，尝试其他方案。

Willison 的经验：代理擅长这种"试很多变体"的任务。

### 案例三：依赖升级

**场景：** 需要升级多个过时的依赖

**目标：** 升级所有依赖到最新版本，所有测试通过

**关键前提：**
> "If your test suite is solid an agentic loop can upgrade them all for you and make any minor updates needed to reflect breaking changes."
>
> — Simon Willison

注意"test suite is solid"这个前提——测试是代理循环有效运行的基石。

### 案例四：容器大小优化

**场景：** Docker 容器太大

**目标：** 在测试通过的前提下最小化镜像大小

代理循环：尝试不同基础镜像 → 迭代 Dockerfile → 验证测试通过 → 检查镜像大小 → 重复。

### 案例五：技术选型原型

**场景：** 评估 Redis 是否适合作为活动流的存储

**Willison 的方法：**
> "The best way to make confident technology choices is to prove that they are fit for purpose with a prototype."

让代理构建一个模拟系统并运行负载测试，根据结果做出决策。由于原型成本几乎为零，可以同时测试多个方案。

## 四、并行代理实战

### 4.1 Willison 的并行代理模式

来源：Willison《Embracing the parallel coding agent lifestyle》

| 模式 | 说明 | 风险 | 审查成本 |
|------|------|------|----------|
| **概念验证研究** | 快速构建原型验证技术方案 | 低（代码不保留） | 无需审查 |
| **"这个怎么工作的？"** | 追踪代码路径生成解释 | 低（只读操作） | 低 |
| **小型维护任务** | 修复 warnings、deprecation 等 | 低 | 低 |
| **精确定义的工作** | 从你的详细规格出发实现 | 中 | 低（你定义了规格） |
| **"派出侦察兵"** | 给代理困难任务只为了解其处理方式 | 低 | 无需审查 |

### 4.2 并行代理的注意事项

- **确保各代理操作的文件不重叠**——否则会产生合并冲突
- **使用 Git 分支或 worktree 隔离各代理的工作**
- **审查每个代理的输出**——代理并行运行不意味着可以跳过审查
- **从你的规格说明出发**——代码源自你自己的规格时，审查成本大幅降低

Willison 的诚实反思：

> "I was pretty skeptical about this at first. AI-generated code needs to be reviewed, which means the natural bottleneck on all of this is how fast I can review the results."

并行的瓶颈不在于代理的速度，而在于你审查的速度。因此，降低审查成本的关键是：
1. 让变更尽可能小（多个小 PR）
2. 让规格尽可能清晰（你定义了目标，审查变成"确认是否符合规格"）

### 4.3 Josh Bleecher Snyder 的"派出侦察兵"模式

来源：《The 7 Prompting Habits of Highly Effective Engineers》

> "Send out a scout. Hand the AI agent a task just to find out where the sticky bits are, so you don't have to make those mistakes."

给代理一个困难任务，不打算保留其代码，只是了解它修改了哪些文件、如何处理问题。这是一种低成本的探索方式。

### 4.4 Peter Steinberger 的经验

来源：Peter Steinberger《Just Talk To It—the no-bs Way of Agentic Engineering》

详细描述了围绕 Codex CLI 的完整工作流程。Willison 推荐这篇文章作为进一步参考。

## 五、凭据管理实战

### 5.1 Willison 的 Fly.io 案例

来源：Willison《Designing Agentic Loops》

Willison 在调查 Fly.io 冷启动性能时，需要让 Claude Code 直接编辑 Dockerfiles、部署到 Fly 并测量启动时间。

他的凭据管理方案：
1. 在 Fly 上创建了一个专用组织
2. 为该组织设置了 $5 的预算上限
3. 生成了仅限于该组织的 API 密钥
4. 将密钥交给 Claude Code

这样即使代理失控，经济损失也被限制在 $5 以内。

### 5.2 Willison 的凭据管理两条规则

1. **优先使用测试/预发布环境的凭据**——任何破坏都在可控范围内
2. **如果凭据能花钱，设置严格的预算上限**

## 六、失败处理

### 6.1 常见失败模式

| 失败模式 | 原因 | 处理方式 |
|----------|------|----------|
| **无限循环** | 代理反复尝试同一方案 | 设置最大迭代次数或 Token 预算 |
| **方向错误** | 代理理解错了目标 | 暂停，重新描述目标，增加约束 |
| **破坏性变更** | 代理删除或重写了重要代码 | 使用 Git 版本控制 + 检查点 |
| **幻觉** | 代理生成了不存在的 API 调用 | 依赖测试和 Linting 捕获 |
| **成本爆炸** | 代理消耗过多 Token | 设置 Token 预算，选择合适的模型 |
| **上下文过载** | 代理在长会话中丢失上下文 | 使用 `/compact` 或重新开始会话 |

### 6.2 人类介入点

以下情况应暂停代理并介入：
- 代理连续 N 次迭代无进展
- 代理的修改影响到核心架构
- 代理遇到了需要业务判断的情况
- 代理的修改影响安全相关代码
- 代理尝试执行你没有预见的操作
