# 测试策略与质量保证

> **核心来源：**
> - Simon Willison《Agentic Engineering Patterns》TDD / First Run the Tests / Agentic Manual Testing 章节

> - Claude Code 官方文档《Best Practices》

## 一、测试在 Agentic Engineering 中的核心地位

> "The single biggest differentiator between agentic engineering and vibe coding is testing."
>
> — Addy Osmani

测试不仅是质量保证手段，更是**代理有效运行的前提条件**。没有测试，代理无法判断自己的修改是否正确，也无法在循环中迭代到正确方案。

---

## 二、测试驱动开发（TDD）与代理

### 2.1 Red/Green TDD 模式

来源：Willison《Red/green TDD》

**流程：**

1. **Red（写失败测试）：** 编写一个描述期望行为的测试，确保它失败
2. **Green（让测试通过）：** 给代理一个清晰目标——"让这个测试通过"
3. **Refactor（重构）：** 审查并优化代码

**为什么 TDD 与代理特别配合：**

- 测试为代理提供了精确的、无歧义的成功标准
- 代理可以在循环中迭代直到测试通过
- 测试先于代码意味着代理不需要"猜测"正确的实现
- 代理生成的代码往往更简洁——只需满足测试，不会过度设计
- 测试结果说了算——代理无法"宣称完成"来蒙混过关

### 2.2 先运行测试

来源：Willison《First Run the Tests》

> 在让代理做任何修改之前，先运行测试确保基线是通过的。

这为后续的代理迭代提供了一个可靠的起点，也确保代理不会因为已有的失败测试而产生混淆。如果基线不是绿色的，代理会把"测试已失败"误认为是自己的修改导致的。

### 2.3 实际效果

Willison 的经验表明，TDD 模式下代理生成的代码：
- **更简洁**——不包含不必要的复杂度（只需满足测试）
- **更可靠**——测试通过 = 代码满足规格
- **更少幻觉**——代理无法"宣称完成"，测试结果说了算

## 三、测试金字塔

在 Agentic Engineering 中，测试金字塔仍然适用，但每一层都有代理可以辅助的方面：

| 测试层 | 传统角色 | 代理的角色 |
|--------|---------|-----------|
| **单元测试** | 手动编写 | 代理可以高效生成单元测试用例；TDD 模式下代理实现代码以通过测试 |
| **集成测试** | 手动编写 + 手动设置环境 | 代理可以设置测试环境、生成测试数据、编写测试夹具 |
| **端到端测试** | 手动 + Selenium/Cypress | 代理可以使用 Playwright 等工具进行 UI 自动化测试 |
| **手动测试** | 人工执行 | 代理可以执行"代理式手动测试"（见下文） |

Claude Code 的官方最佳实践进一步强化了测试的核心地位：

> "The single highest-leverage thing you can do to improve Claude Code's effectiveness on a task is to run tests. Claude Code can then use test results to iterate on its changes."
>
> — Anthropic《Claude Code Best Practices》

## 四、代理辅助的测试生成

### 4.1 为现有代码生成测试

代理可以帮助为现有代码生成测试：

> "If you don't have one [test suite] yet, thankfully LLMs are great for accelerating the process of putting one together."
>
> — Simon Willison

代理可以：
- 分析函数签名和类型注解生成测试用例
- 根据错误处理逻辑生成边界条件测试
- 基于代码注释和文档字符串生成行为测试
- 参考项目中的现有测试模式保持一致性

### 4.2 代理式手动测试

来源：Willison《Agentic Manual Testing》

代理可以执行"手动"测试的自动化版本——不是替代人类的视觉判断，而是执行那些机械的、重复的手动测试步骤。

**机制：**
- 让代理模拟用户交互流程
- 使用浏览器自动化（Playwright）进行 UI 测试
- 让代理记录测试过程中的发现

**工具：**

| 工具 | 用途 | 来源 |
|------|------|------|
| **Playwright** | 浏览器自动化测试 | Willison 推荐 |
| **Showboat** | 让代理记录笔记的工具 | Willison 推荐 |
| **curl / httpie** | API 端点测试 | 通用 |
| **httpx** | Python HTTP 测试 | 通用 |

**局限性（Willison 明确指出）：**
- 代理可能遗漏只有人类能发现的交互问题
- 对视觉布局问题的判断不如人类精确
- 可能无法模拟所有边缘用户行为

### 4.3 截图验证

Claude Code 的 Computer Use 能力可以截取屏幕截图，验证 UI 变更是否符合预期。这对于前端开发特别有价值。

## 五、代码审查策略

### 5.1 人工审查

> "You review that code with the same rigor you'd apply to a human teammate's PR. If you can't explain what a module does, it doesn't go in."
>
> — Addy Osmani

### 5.2 代理辅助审查

- 使用代理进行初步审查（标识明显问题）
- 人工进行最终审查（架构决策、业务逻辑）
- Claude Code 的 GitHub Code Review 功能可以自动审查 PR

### 5.3 审查代理生成的 PR 描述

来源：Willison《Anti-patterns》

> "Agents write convincing looking pull request descriptions. You need to review these too! It's rude to expect someone else to read text that you haven't read and validated yourself."

代理不仅生成代码，还生成 PR 描述——这些描述也需要审查。它们看起来很专业，但可能包含不准确的信息。

## 六、CI/CD 中的代理

### 6.1 代理生成代码的 CI 检查

代理生成的代码应与传统代码通过完全相同的 CI 检查：

```
代码提交 → 单元测试 → 类型检查 → Linting → 构建 → 安全扫描 → 人工审查
```

### 6.2 代理作为 CI 工具

代理可以用于加速 CI/CD 流水线本身：
- 自动修复 CI 中发现的 linting 错误
- 自动更新依赖并验证兼容性
- 自动生成变更日志
- Claude Code 非交互模式（`-p`）直接嵌入 GitHub Actions / GitLab CI

### 6.3 GitHub Actions 集成

Claude Code 支持在 GitHub Actions 中运行，可以自动化：
- PR 审查
- Issue 分类
- 代码质量检查
- 依赖安全扫描

## 七、质量保证清单

### 7.1 代码质量

- [ ] 是否有全面的自动化测试套件？
- [ ] 测试是否在代理迭代之前通过（基线绿色）？
- [ ] 是否有类型检查和 Linting？
- [ ] 是否有 CI 流水线自动运行测试？

### 7.2 审查流程

- [ ] 所有代理生成的代码是否经过人工审查？
- [ ] 是否审查了代理生成的 PR 描述？
- [ ] PR 变更是否足够小以便高效审查？

### 7.3 验证方法

- [ ] 是否有预发布环境进行验证？
- [ ] 是否有安全扫描？
- [ ] 是否有依赖漏洞检查？
- [ ] 对于 UI 变更，是否有截图验证？

### 7.4 代理特定

- [ ] 代理是否在沙箱中运行？
- [ ] 是否设置了 Token 预算？
- [ ] 是否有最大迭代次数限制？
- [ ] 是否有停止条件配置？
