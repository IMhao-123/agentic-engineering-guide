## 一、术语演变时间线

### 2025.02.06：Vibe Coding 的诞生

**Andrej Karpathy** 在 X（原 Twitter）上发表推文，首次使用 "Vibe Coding" 一词：

> "There's a new kind of coding I call 'vibe coding', where you fully give in to the vibes, embrace exponentials, and forget that the code even exists."
>
> — Andrej Karpathy（2025.02.06）

这一定义迅速在社区传播。Karpathy 是 OpenAI 联合创始人，曾任 Tesla AI 总监，在 AI 社区具有巨大影响力。

**《纽约时报》《卫报》等主流媒体对此进行了报道，进一步扩散了这个术语。**

### 2025.02 - 2025.06：编码代理工具的涌现

编码代理工具——能够同时生成**和**执行代码的 AI 工具——开始密集发布：

| 工具 | 发布方 | 发布时间 | 说明 |
|------|--------|----------|------|
| Claude Code | Anthropic | 2025.02 | 随 Claude 3.7 Sonnet 一起发布的终端编码代理 |
| Codex CLI | OpenAI | 2025.04 | OpenAI 的命令行编码代理 |
| Gemini CLI | Google | 2025.06 | Google 的命令行编码代理 |

**关键转变：** AI 不再仅仅是"生成代码建议"，而是可以**在循环中执行、测试和迭代代码**。这个技术转变是术语演变的底层驱动力。

### 2025.03.19：Vibe Coding 权威定义

**Simon Willison** 发表《Not all AI-assisted programming is vibe coding (but vibe coding rocks)》：

> "When I talk about vibe coding I mean **building software with an LLM without reviewing the code it writes**."
>
> — Simon Willison

Willison 指出了术语泛化的问题：

> "I'm concerned that the definition is already escaping its original intent. I'm seeing people apply the term 'vibe coding' to all forms of code written with the assistance of AI."
>
> — Simon Willison

他明确了 Vibe Coding 与 AI 辅助编程的区别，并肯定了 Vibe Coding 的价值——让非程序员也能构建工具。

### 2025.09.30：Agentic Loop 概念的确立

**Simon Willison** 发表《Designing Agentic Loops》，首次系统性地定义了"代理循环设计"概念：

> "A critical new skill to develop is designing agentic loops. One way to think about coding agents is that they are brute force tools for finding solutions to coding problems. **If you can reduce your problem to a clear goal and a set of tools that can iterate towards that goal a coding agent can often brute force its way to an effective solution.**"
>
> — Simon Willison

**核心定义：**

> "My preferred definition of an LLM agent is something that **runs tools in a loop to achieve a goal**."
>
> — Simon Willison

这为 Agentic Engineering 提供了技术基础——代理循环是编码代理的核心执行模式。

### 2025.10.05：并行代理工作模式

**Simon Willison** 发表《Embracing the parallel coding agent lifestyle》：

> "I'm increasingly hearing from experienced, credible software engineers who are running multiple copies of agents at once, tackling several problems in parallel and expanding the scope of what they can take on."
>
> — Simon Willison

他记录了同时运行多个编码代理的工作模式，为 Agentic Engineering 提供了实践范例。

### 2025.10.07：Vibe Engineering 的提出

随着 "Vibe Coding" 被越来越宽泛地使用，**Simon Willison** 在《Vibe Engineering》中提出需要一个新术语：

> "I feel like vibe coding is pretty well established now as covering the fast, loose and irresponsible way of building software with AI. **This leaves us with a terminology gap: what should we call the other end of the spectrum?**"
>
> — Simon Willison

他提议使用 "Vibe Engineering" 来区分专业端的 AI 辅助开发，并详细列出了 12 项 LLM 实际上奖励的高级软件工程实践。

**但这个术语有一个问题：** "Vibe" 一词带有"随意性"的内涵，在专业场景中不够严肃。

### 2026.02.04：Agentic Engineering 的确立

**Addy Osmani** 发表《Agentic Engineering》，系统性地：

1. 批评了 "Vibe Coding" 被过度泛化使用的问题
2. 回顾了 Willison 的 "Vibe Engineering" 提议
3. 引用了 **Karpathy 建议使用 "Agentic Engineering"** 的推文
4. 系统定义了 Agentic Engineering 的工作方式、与 Vibe Coding 的区别

**Osmani 的核心定义：**

> "**Vibe coding = YOLO. Agentic engineering = AI does the implementation, human owns the architecture, quality, and correctness.**"
>
> — Addy Osmani

### 2026.02.23：Agentic Engineering Patterns 指南

**Simon Willison** 正式将术语更新为 "Agentic Engineering"，开始维护《Agentic Engineering Patterns》：

> "It looks like the term 'Agentic Engineering' is coming out on top for this now."
>
> — Simon Willison

> "I've started a new project to collect and document **Agentic Engineering Patterns**—coding practices and patterns to help get the best results out of this new era of coding agent development we find ourselves entering."
>
> — Simon Willison

这个指南以类似于经典《设计模式》的方式组织，按 1-2 章/周的频率持续发布。

## 三、为什么需要一个新术语

### 3.1 "Vibe Coding" 的泛化问题

Osmani 精准地指出了术语混乱带来的危害：

> "People now use it to describe everything from a weekend hack to a disciplined engineering workflow where AI agents handle implementation under human oversight. **These are fundamentally different activities, and conflating them is causing real confusion—and real damage.**"
>
> — Addy Osmani

当人们无法区分"周末的快速原型"和"有纪律的代理辅助生产开发"时，企业和技术管理者就无法做出正确的技术决策。

### 3.2 Willison 对 "Vibe Engineering" 的反思

> "It looks like the term 'Agentic Engineering' is coming out on top for this now."
>
> — Simon Willison（2026.02.23）

"Vibe" 一词带有"随意性"的内涵，在专业场景中不够严肃。Willison 最终接受了 "Agentic Engineering" 作为更好的术语。

### 3.3 "Agentic Engineering" 的优势

| 维度 | 说明 |
|------|------|
| **准确描述行为** | 你在编排（orchestrating）AI 代理，代理执行实现 |
| **职业可读性** | 可以在技术会议、招聘、管理层报告中使用 |
| **天然划界** | 术语本身强制了与 Vibe Coding 的区分 |
| **技术基础关联** | "Agentic" 直接关联到 "Agentic Loop"——技术核心 |

## 四、范式跃迁的本质

### 4.1 技术转变是驱动因素

这场范式跃迁的真正驱动力不是某个术语的命名，而是**编码代理工具的出现**。

当 AI 只能生成代码建议（如 GitHub Copilot 的补全功能）时，开发者仍然需要手动输入每一行代码，"Vibe Coding" 和 "Agentic Engineering" 之间的区别并不明显。

但当 AI 可以**在循环中自主执行、测试和迭代代码**时，开发者面临的选择变得截然不同：
- **你可以选择不审查** → Vibe Coding
- **你可以选择审查并建立工程纪律** → Agentic Engineering

### 4.2 工具的成熟放大了差距

随着编码代理能力的提升（更强的推理、更好的工具使用、更长上下文），两种方式的产出差距越来越大：
- Vibe Coding 的产出：越来越庞大但不可理解的代码
- Agentic Engineering 的产出：越来越高质量的生产级软件

### 4.3 专业门槛在上升而非下降

> "Agentic engineering isn't easier than traditional engineering—it's a different kind of hard."
>
> — Addy Osmani

这不是"AI 让编程变得简单"的故事，而是"AI 放大专业能力"的故事。经验丰富的工程师获得的力量倍增器远大于新手——因为 Agentic Engineering 需要的技能（架构设计、代码审查、测试策略、安全意识）本身就是资深工程师的优势领域。
